# 测试结果双维度优化

## 问题分析

当前测试结果只有一个 `status` 字段（`success` / `fail`），存在以下问题：

1. **语义混淆**：`success` 同时表示"执行完成"和"验证通过"，`fail` 可能是"设备掉线"也可能是"验证不通过"
2. **失败原因不透明**：用户无法从列表页直接区分"执行中断"和"结果不达标"
3. **步骤不可见**：失败时虽然 reporter_node 补充了步骤摘要到 conclusion，但步骤时间线本身数据已存在，只是 UI 展示不够突出
4. **无验证清单**：Planner 生成了 `verification` 验证条件，但执行结果没有逐条对照

## 目标数据结构

```
执行状态 (execution_status)     测试结论 (test_verdict)
─────────────────────────────   ──────────────────────────
completed  - 执行正常完成        passed    - 验证条件全部达标
exhausted  - 步骤数耗尽          failed    - 验证条件有不达标项
error      - 意外错误/设备掉线   inconclusive - 无法判定（执行未完成）
cancelled  - 用户取消
device_offline - 设备未连接
```

组合示例：
- `completed` + `passed` = 绿色 ✅ 执行完成，测试通过
- `completed` + `failed` = 红色 ❌ 执行完成，但有验证项不达标
- `exhausted` + `inconclusive` = 黄色 ⚠️ 步骤用完，无法确认结果
- `error` + `inconclusive` = 灰色 ⚙️ 执行中断，无法验证

## Task 2: 改进 reporter_node 判定逻辑

**文件**: `agents/graph.py` — `reporter_node()`

### 2.1 确定 execution_status

```python
def _determine_execution_status(state: dict) -> str:
    status = state.get("status", "")
    conclusion = state.get("conclusion", "")
    
    if status == "cancelled":
        return "cancelled"
    if status == "device_offline":
        return "device_offline"
    
    # Agent 输出 DONE → completed
    done, abort = _detect_termination(conclusion)
    if done:
        return "completed"
    if abort:
        if "MAX_TURNS" in conclusion:
            return "exhausted"
        # Agent 主动 ABORT（如"无法找到目标元素"）仍算 completed：
        # 执行流程正常结束了，只是验证结论由 test_verdict 决定（failed）
        return "completed"
    
    # 没有 DONE 也没有 ABORT → 可能被路由强制终止
    history = state.get("step_history", [])
    if len(history) >= 12:
        return "exhausted"
    
    return "error"
```

### 2.2 确定 test_verdict — 基于 assert_verification 工具

**核心改动**：不再从 DONE 文本逆向提取验证结果，而是让 Agent 在执行过程中主动逐条报告。

#### 2.2.1 新增 assert_verification 工具

**文件**: `tools/__init__.py`

验证结果挂在 `ToolContext._verifications` 上。`ToolContext` 是服务级单例，每次测试执行前由 orchestrator 主动调用 `ctx._verifications.clear()` 清空（见 Task 4.0），避免并发冲突和残留。

```python
@tool
def assert_verification(condition: str, result: str, detail: str = "") -> str:
    """逐条报告验证条件的结果。
    
    Args:
        condition: 验证条件描述（对应 goal.verification 中的某一条）
        result: "passed" 或 "failed"
        detail: 补充说明（可选）
    
    当 result="failed" 时自动截图，路径存入 verification 记录供前端展示。
    """
    ctx = get_tool_context()
    if ctx:
        if not hasattr(ctx, '_verifications'):
            ctx._verifications = []
        # 失败时自动截图佐证
        shot_path = ""
        if result == "failed" and ctx.device:
            try:
                import os, re as _re
                from datetime import datetime
                os.makedirs("storage/screenshots", exist_ok=True)
                safe_cond = _re.sub(r'[^\w\u4e00-\u9fff-]', '_', condition[:30])
                shot_path = f"storage/screenshots/verify_{datetime.now().strftime('%Y%m%d_%H%M%S_%f')}_{safe_cond}.png"
                ctx.device.screenshot().save(shot_path)  # screenshot() 返回 PIL Image
            except Exception:
                pass
        ctx._verifications.append({
            "item": condition,
            "result": result if result in ("passed", "failed") else "unknown",
            "detail": detail,
            "screenshot": shot_path,  # 失败时不为空，前端展示缩略图
        })
    return f"记录完成: {condition} → {result}"
```

**注意**：不使用模块级全局变量（`_verification_assertions`），因为：
- 两次测试并发执行时会共享同一个列表
- 测试结束后不清空，下次测试会残留上次的断言

#### 2.2.3 reporter_node 收集验证结果

直接从 ToolContext 读取结构化数据，不依赖正则扫描 observation 文本：

```python
def _collect_verification_results(goal: dict) -> tuple[str, list]:
    """从 ToolContext 中收集 assert_verification 的结构化结果。"""
    ctx = get_tool_context()
    assertions = getattr(ctx, '_verifications', []) if ctx else []
    
    # 如果 Agent 没有调用 assert_verification，回退到简单推断
    if not assertions:
        verification_items = goal.get("verification", [])
        if verification_items:
            # 有验证条件但 Agent 没逐条报告 → 标记为 unknown
            assertions = [{"item": v, "result": "unknown"} for v in verification_items]
        else:
            return "passed", []
    
    # 汇总判定
    passed = sum(1 for a in assertions if a["result"] == "passed")
    failed = sum(1 for a in assertions if a["result"] == "failed")
    
    if failed > 0:
        verdict = "failed"
    elif passed == len(assertions):
        verdict = "passed"
    else:
        verdict = "inconclusive"
    
    return verdict, assertions
```

**为什么不用 regex 扫 observation**：
- observation 可能被截断（500 字符），导致 "已记录: xxx → passed" 被截掉
- Agent 的描述措辞可能有微小变化（如 "已完成记录" vs "记录完成"）
- ToolContext 是进程内结构化传递，零解析成本

```python
    execution_status = _determine_execution_status(state)
    test_verdict, verification_results = _collect_verification_results(
```

### 2.4 步骤级成功/失败着色

| status | 颜色 | 含义 |

### 3.1 新增迁移方法

```python
def _migrate_v3_columns(self) -> None:
    """V3 迁移：为 test_runs 新增 execution_status / test_verdict / verification_json 列。"""
    for col, typedef in [
        ("execution_status", "TEXT DEFAULT ''"),
        ("test_verdict", "TEXT DEFAULT ''"),
        ("verification_json", "TEXT DEFAULT '[]'"),
    ]:
        try:
            self._conn.execute(f"ALTER TABLE test_runs ADD COLUMN {col} {typedef}")
        except sqlite3.OperationalError:
            pass
    self._conn.commit()
```

### 5.2 新增辅助函数

```javascript
const execStatusMap = {
  completed:     { label: '已完成', type: 'success' },
  exhausted:     { label: '步骤耗尽', type: 'warning' },
  error:         { label: '异常中断', type: 'danger' },
  cancelled:     { label: '已取消', type: 'info' },
  device_offline:{ label: '设备离线', type: 'info' },
};
const verdictMap = {
  passed:       { label: '通过', type: 'success' },
  failed:       { label: '未通过', type: 'danger' },
  inconclusive: { label: '待确认', type: 'warning' },
};

function execStatusLabel(s) { return (execStatusMap[s] || execStatusMap.error).label; }
function execStatusType(s)  { return (execStatusMap[s] || execStatusMap.error).type; }
function verdictLabel(s)    { return (verdictMap[s] || verdictMap.inconclusive).label; }
function verdictType(s)      { return (verdictMap[s] || verdictMap.inconclusive).type; }
```
