### 0.5 最终 P1 建议（复跑前事实校验，v3.2 未明确）

> **前置说明（开发阶段）**：当前仍处开发期，**不存在跨数据源 Merge 场景**，故 R20/R21 以「轻量事实校验 + LLM 决策」为上限，不做复杂数据合并/回填逻辑。

**已核对的 `goal_json` 实际结构**（来自 planner 输出与运行日志，非假设）：
```json
{"goal":"一句话目标","app_package":"com.android.settings","app_name":"Settings",
 "target_pages":["设置主页","WIFI设置页"],          // List[str]，模糊页面名，无 Activity 映射
 "verification":["WIFI开关可从关到开","WIFI开关可从开到关"], // List[str]，验证条件文本
 "hints":["左侧导航需滚动找到WLAN","切换后等1秒检查"]}
```
→ `target_pages` 是**页面名字符串**（非 `{"package","activity"}`），静态无法映射到 Android Activity；`ctx.perceiver` 需调 `perceive()` 才返回 `PageUnderstanding`，**无 `current_page.pages` 直接属性**，页面名存在性只能由 LLM 在真实导航中发现。

| # | 建议 | 处置（修正后） |
|---|---|---|
| R20 | 复跑前应用可启动性检查（P1，预防「改版/卸载后复跑无限空转」） | ✅ 收窄为**仅校验 `app_package` 是否可启动**（adb `pm list packages` 级别轻量检查，经 device/adb 层，不依赖 perceiver）。**代码不直接中止**——检查失败只把事实「应用 `<pkg>` 在当前设备未安装/无法启动」**注入 system prompt**，由 LLM 自行决定调用 `report_done(status="abort",...)` 还是尝试替代路径。页面名是否存在**不静态校验**，留给 LLM 在 steps 中感知 |
| R21 | 复跑前验证条件可满足性提示（P1，同上） | ✅ 不主动做元素快照检测（元素 id 不在 `goal_json` 中，无法静态定位）。改为：将 `verification` 列表**原样写入 system prompt 作为固定验证契约**；若 LLM 在 steps 中经 `perceive()` 发现**全部**验证元素缺失，由其自主 `report_done(status="abort", summary="验证元素 X 已不存在")`，**部分缺失**则由 LLM 在 steps 中二次确认是否继续。代码只提供契约与感知结果，决策权在 LLM |

**关于 `report_done(status="abort")` 格式（已核对受支持）**：`agent_node` 将 `status` 置为 `"fail"`，`orchestrator` 经 `_detect_termination` 从 `conclusion` 解析 `is_abort`（nodes.py/orchestrator.py 已有该分支）。但**代码路径不主动调用**——事实注入后由 LLM 在 steps 中自主决定终止方式，完全符合 §0「代码给 ground truth，LLM 做决策」。

**设计原则契合**：R20/R21 不违反《综合评审与问题分类》原则，反而强化「代码只负责稳的地基」——由代码提供**可核实的事实**（应用是否可启动、验证契约、感知结果），让 LLM 在其上做决策，而不是让 LLM 凭空感知页面改版（易误判/空转）。属 **P1 预防型增强**，落地优先级低于 P0 主链路（复用/并发/迁移），但应随复跑入口一并实现。

**刷新边界（无需修，仅记一笔）**：见 §2.4 多窗口同步段末——刷新后旧 run 的 `plan_review` 待确认状态成为孤儿，无法 resume，影响极低。

### 0.6 实现前代码核对补充（v3.3 代码 review 结论，必读）

> 把文档方案逐条对照 `graph.py`/`state.py`/`nodes.py`/`orchestrator.py`/`relational.py`/`server.py` 后，核心设计（route_start、state 字段预填、busy-guard、迁移、COALESCE）与代码吻合，但发现 **1 个真 bug 级漏洞 + 5 个实现边界**，实现时必须处理。

**🔴 R1（必修）：复跑跳过 `planner` 会泄漏 RAG 缓存 / 计数器**
- `agents/nodes.py` 的 `planner_node` 开头（约 L218-232）会重置一批 **ctx 级 run-scoped 属性**：
  ```python
  ctx._rag_query_cache = {}; ctx._run_tag = (...)
  ctx._rag_query_count = 0; ctx._rag_same_app_count = 0
  ctx._rag_cross_app_count = 0; ctx._rag_empty_hit_count = 0
  ```
- 而 `orchestrator._reset_run_scoped(ctx)`（约 L20-35）**只重置** `_verifications` / `_deterministic_checks` / `_token_usage`，**没有**上面这 6 个 RAG 属性。
- `get_tool_context()` 返回**模块级单例 ctx，跨 run 复用**。普通 run 因 `planner` 先跑、RAG 缓存被清，无问题；但**复跑跳过 planner → 这 6 个属性不重置 → 上次 run 的 RAG 缓存/计数直接泄漏进复跑**，导致复跑 RAG 命中率统计失真、甚至用旧缓存给出错误知识。
- **修复（落地到 §3.3 / §7 step4）**：把上述 6 个 RAG 属性重置**从 `planner_node` 搬到 `_reset_run_scoped`**（它本就被 `start`/`start_stream` 每次调用），这样普通 run 与复跑都覆盖，`planner_node` 内的重置变成冗余但无害。`_reset_run_scoped` 是 ctx 字段的唯一定点，复跑必须同时重置 **state 字段（§3.3 initial_state 预填）** 与 **ctx 字段（_reset_run_scoped）** 两类——原 §3.3 把二者混为一谈，需在实现时区分。

**🟡 R2–R6（实现边界，非 bug，但易踩）**
- **R2 WS `rerun`/`run_case` 必须走 sync `start`，不能改 `start_stream`**：现有 `run` 分支是 `await asyncio.to_thread(orchestrator.start, ...)`，实时时间线来自 `orchestrator._emit`→`broadcast_sync` 的逐条广播（run_started/status/…）+ 末尾 `{type:"result"}`。故「时间线与普通 run 完全一致」成立的前提是 `rerun`/`run_case` 也用 **sync `start` + 广播**；若实现者顺手改成 `start_stream` 反而打断事件流（见 §3.4）。
- **R3 `route_start` 不在 resume 时重算（安全，但建议加锁测试）**：条件边 `START→route_start` 仅在**首次 invoke** 求值；`plan_review` 中断后 `resume()` 从中断节点续跑，不会重跑 START，即使此时 `goal_description` 已写好也不会被误路由跳过 `plan_review`。属 LangGraph 设计行为，无 bug，但建议补 `test_route_start_only_on_initial_invoke` 锁死该语义，防未来升级改变。
- **R4 `_calc_budget_from_state` 对复跑的依赖**：`route_after_agent` 用 `_calc_budget_from_state(state)` 算 `max_agent_iterations`，须能从复跑已设的 `state["goal_description"]` 读到 goal，预算才正确。实现时确认它读 `state["goal_description"]`，并单测覆盖复跑的 `max_iterations`。
- **R5 `test_cases` CRUD 加在 `SqliteBackend`，不是 ABC**：`RelationalBackend` 是抽象基类（只声明 `execute/insert/select/upsert/count`）；`record_test_run`/`list_test_runs` 都是 `SqliteBackend` 具体方法。`record_case_run`/`list_test_cases`/`get_test_case`（以及 `test_cases` 建表语句）必须同样加到 `SqliteBackend`，不要往 ABC 塞具体逻辑。**v3.3 不实现 `_migrate_columns`**——`goal_json`/`run_type`/`source_run_id` 三列直接写进 `_ensure_tables` 的 `CREATE TABLE`（L91-93），首次启动或清空 `storage/test_history.db`（含 `-wal`/`-shm`）后由 SQLite 重建。
- **R6 普通 run 的 `_run_type`/`_source_run_id` 为可选冗余**：`reporter_node` 用 `state.get("_run_type","normal")`/`state.get("_source_run_id")`，key 缺失即默认 `"normal"`/`None`，故普通 run 即使 `initial_state` 不写这两个字段也正确——§3.3 列了无妨，实现时不必强求。

### 2.4 并发串行化（满足"等当前结束才能跑"）

```
前端：executing=true 时，所有「复跑/运行/工作台执行」按钮禁用 + 提示「请等待当前运行结束」。
后端：orchestrator.start/start_stream 入口检查 self._active_runs：
      若非空 → 直接返回 {"status":"busy","message":"已有运行在进行，请等待其结束"}
      （device 单设备，禁止并发；避免图状态串扰）
```
> v3 用「前端锁 + 后端 busy-guard」实现串行，不做后台 FIFO 队列（降低复杂度；后续如需"自动排队"再扩展）。
> 多窗口同步：后端 `run_started`/`run_stopped` 经 `broadcast_sync` 广播**所有** WS 连接；前端任一窗口收到即设/清全局 `executing`，故 B 窗口也能感知 A 窗口的运行中状态（并被后端 busy-guard 兜底拒绝）。
> 刷新边界（无需修，仅记一笔）：`plan_review` 的 `interrupt()` 经 `GraphInterrupt` 被 `except` 捕获并 `yield` 给前端后，`finally` 中 `_cleanup_run(thread_id)` 已清空 `_active_runs`。若此时**页面刷新**：`GET /api/runs/active` 返回 `{"active": false}`，前端 `executing=false`、按钮恢复；而旧 run 在 checkpointer（MemorySaver）里的 `interrupt` 待确认状态成为**孤儿**，无法再 `resume`（用户只能新建 run）。影响极低——刷新后旧 `plan_review` 对话框已丢失，不存在"误操作旧 run"场景；唯一代价是 MemorySaver 多一条孤儿状态（进程重启即清）。故不修。

### 2.5 复跑 / 运行 时序

```
单条复跑（WS）：
T+0  点「复跑」→ 前端读 report.goal_json（busy 时禁用）
T+0  WS {type:"rerun", run_id, goal} → orchestrator.start(goal_description=goal, reuse_plan=True, run_type="rerun", source_run_id=run_id)
T+0  route_start 检测 goal 非空 → 直达 agent（跳过 planner+plan_review）
T+x  reporter 写 test_runs（goal_json + run_type='rerun' + source_run_id）
```

```python
def _migrate_columns(self) -> None:
    cols = {r[1] for r in self._conn.execute("PRAGMA table_info(test_runs)")}
    needed = {
        "goal_json":     "TEXT NOT NULL DEFAULT '{}'",
        "run_type":      "TEXT NOT NULL DEFAULT 'normal'",
        "source_run_id": "TEXT",
    }
    for name, ddl in needed.items():
        if name not in cols:
            try:
                self._conn.execute(f"ALTER TABLE test_runs ADD COLUMN {name} {ddl}")
            except Exception as exc:   # 权限/锁表等，记录但不中断启动
                logger.error("migrate column %s failed: %s", name, exc)
    self._conn.commit()
```
> 注意：`ADD COLUMN` 带 `NOT NULL` 时**必须**给 `DEFAULT`，否则旧行无值会报错；`source_run_id` 允许 NULL 故不加 `NOT NULL`。**当前 SqliteBackend 已无此方法**（v3.3 不实现），需要时再补。

```python
def start(self, user_request, app_package="", app_name="", thread_id="",
          goal_description=None, reuse_plan=False,
          run_type="normal", source_run_id=None) -> dict:
    ctx = get_tool_context(); _reset_run_scoped(ctx)
    if getattr(ctx, "device", None) is None:
        return {...device_offline...}
    # F7 并发保护：已有运行在进行则拒绝
    if self._active_runs:
        self._emit("error", {"message": "已有运行在进行，请等待其结束后重试"})
        return {"status":"busy","execution_status":"busy","test_verdict":"inconclusive",
                "verification_results":[],"mode":"run","conclusion":"BUSY: 已有运行在进行",
                "steps":[],"thread_id": thread_id}
    if not thread_id:
        thread_id = f"test-{datetime.now().strftime('%Y%m%d_%H%M%S')}"
    _stop_ev = self._register_run(thread_id); self._attach_stop_event(ctx, _stop_ev)
    self._emit("run_started", {"thread_id": thread_id, "started_at": datetime.now().isoformat()})
    # 复跑预填 planner 原本设置的初始字段（P0 #2）
    initial_state = {
        "user_request": user_request, "app_package": app_package, "app_name": app_name,
        "goal_description": goal_description or {},
        "step_history": [], "messages": [], "step_times": [],
        "started_at": datetime.now().isoformat(),     # reporter 时长必需
        "conclusion": "", "status": "", "_stop_requested": False,
        "budget_violation_count": 0, "llm_call_count": 0,
        "tool_call_400_count": 0, "tool_call_400_rate": 0.0,
        "_rag_injected_once": False, "_rag_last_app_package": "",
        "_knowledge_query_hint_injected": False, "_last_page_app_key": "",
        "_last_clickable_count": 0,
        "_run_type": run_type or "normal", "_source_run_id": source_run_id,  # 内部字段，与 _stop_requested 风格一致
    }
    # 后续 invoke / 结果上报 / 清理 完全复用现有路径
```
`start_stream` 同步增加同款参数与 `initial_state` 预填 + busy-guard。由于 `start_stream` 是 `AsyncIterator`，busy 时**先 `yield` 再 `return`**（不能像 `start` 那样直接 return dict）：
```python
async def start_stream(self, user_request, app_package="", app_name="",
                       thread_id="", goal_description=None, reuse_plan=False,
                       run_type="normal", source_run_id=None):
    ctx = get_tool_context(); _reset_run_scoped(ctx)
    if getattr(ctx, "device", None) is None:
        yield {"type": "error", "content": "Android 设备未连接，请检查 USB/ADB 连接后重试"}
        return
    if self._active_runs:                      # F7 并发保护
        yield {"type": "error",
               "content": "已有运行在进行，请等待其结束后重试"}
        return
    # 以下复用现有流式逻辑；initial_state 同样预填 _run_type/_source_run_id 与 planner 初始字段
    ...
```

> ⚠️ **ctx 字段与 state 字段必须分别重置（见 §0.6 R1）**：复跑跳过 `planner_node`，而 `planner_node` 原本在内部重置的 6 个 RAG 相关 ctx 属性（`_rag_query_cache`/`_run_tag`/`_rag_query_count`/`_rag_same_app_count`/`_rag_cross_app_count`/`_rag_empty_hit_count`）不会被执行 → 单例 ctx 跨 run 泄漏。实现时把这 6 个重置**从 `planner_node` 搬到 `_reset_run_scoped`**（每次 `start`/`start_stream` 都调），state 字段仍按上面 `initial_state` 预填。两类缺一不可。
