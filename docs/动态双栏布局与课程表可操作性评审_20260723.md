# 动态双栏布局与课程表可操作性评审

> 日期：2026-07-23  
> 状态：审核结论已采纳；P0 当前屏幕可见的真实 clickable 索引统一、右侧摘要改写、P2 `split_x <= 45%` 防御约束及屏幕可见性过滤已实施。`click(region=...)` 仍待独立评审。

## 1. 结论摘要

`PageUnderstanding.layout` 应当保留，但不应再承担“页面真实结构”的权威职责。

- `layout` 保留为向后兼容的粗粒度摘要：`single_pane` / `two_pane`。
- `regions` 是面板位置、滚动范围和元素归属的权威来源。
- 动态、互斥的分栏和按 `regions` 滚动是必要改动；它们解决了固定阈值重叠、固定 x 坐标滚动和跨面板误点的真实问题。
- 课程表失败的主因不是右侧没有可点击元素，而是当前“可点击候选”错误地要求元素有非空 `label`。这使无文本、但真实可点击的父课程格没有 index，也不能被 `click(index=n)` 选中。
- 因此，当前“右侧 `actionable=0` / 没有可由 `click()` 定位的元素”的告警结论错误且有害，必须撤回；“不要用左侧编号代替右侧目标”的约束仍然正确。

## 3. 已确认的代码根因与契约违背

以下三个位置均将“可执行”错误收窄为“可点击且 label 非空”：

- `agents/nodes.py`：`page_info` 的 `clickable_elements` 和 `[n]` 渲染条件为 `e.clickable and e.label`。
- `tools/click.py::_exact_clickable_candidates()`：`click(index=n)` 的候选池为 `e.clickable and (e.label or "").strip()`。
- `tools/__init__.py::get_screen_info()`：`indexed_clickables` 同样排除无 label；`mode="clickable"` 还会以 label 去重，空 label 项不会进入输出。

这造成同一页面同时出现两套矛盾事实：完整元素输出会标记无文本父节点为 `[CLICKABLE]`，但面板 `actionable` 统计、全局 `[n]` 与 `click(index=n)` 却假定它不存在。故原文“`[n]` 始终是全局可点击序号”的描述不成立；实际它是“全局**带文本**可点击序号”。

## 4. 已落地且仍然必要的修改

### 4.1 动态、互斥的面板边界

文件：`device/perceiver.py`

`_detect_pane_regions(elements, width, height)` 从横向元素中心间隙推导分隔线，忽略横跨页面的共享 chrome，并产生相接、不重叠的 `[0, split_x]` / `[split_x, width]` 区域。跨分隔线元素归为 `main_content`。判定 `left_navigation` 时，左侧必须至少有 3 个带文本且纵向分布的可点击入口；无文本的课程网格单元不能作为导航证据。

这替代了左 `0%~45%`、右从 `35%` 开始的重叠阈值，仍是必要改动。

### 4.2 动态面板滚动

文件：`tools/device_ops.py`

`scroll_panel()` 已改为读取 `PageUnderstanding.regions`，在目标面板 x 中点滑动，并且将前后新增元素比较限制在相同面板。它修复固定 `22%/72%` 坐标不能可靠滚到课程表的问题。

### 4.3 `layout` 保留为兼容摘要

保留：

```python
layout = "two_pane" if len(regions) == 2 else "single_pane"
```

`layout` 仅表达粗粒度页面形态和兼容分支；实际面板边界、元素归属与滚动依赖 `regions`。`layout="two_pane"` 不能推断某个具体元素可点击，但也不能据此否定右侧已暴露的无文本可点击元素。

### 4.4 已有覆盖

`tests/test_dynamic_pane_layout.py` 覆盖动态分隔位置、边界不重叠、左右唯一归属及共享标题归 `main_content`；关联聚焦测试此前为 `20 passed`。

## 5. 已实施：P0 真实 clickable 索引契约

### 5.1 统一候选池

以下位置现已统一以 `element.clickable is True` 作为全局候选池条件，不再以 label 为空过滤：

1. Agent `page_info` 的全局 `[n]`；
2. `get_screen_info()` 的 `indexed_clickables` 与面板统计；
3. `click(index=n)` 的精确候选池。

因此，无文本课程格也会获得全局 `[n]`，并可通过 `click(index=n)` 直接执行 bounds 点击（`label` 仅作为 Agent 的意图描述，不参与定位）。`[n]` 的正式语义已修正为“当前屏幕可见的真实可点击元素序号”。

> 注意：P0 解决了“点不到”的问题，但 Agent 仍无法从 `[n]` 本身知道“哪个是第一节空课程格”。若课程表目标需要精确定位到某行某列，可再讨论基于视觉/行列表头的辅助定位；本例先依赖 bounds 空间推断。

### 5.2 展示上限、导航锚点与大规模无文本元素

`page_info` 代表当前屏幕可见内容。perceiver 已在 `_heuristic_understand` 阶段过滤掉完全在屏幕外的元素，因此索引池只覆盖**当前屏幕内**的真实 `clickable`。需要滚动才能看到的内容，应在滚动后重新 perceive，而不是一次性列进当前视野。

由于单个屏幕内仍可能包含大量可点击元素（例如课程表一屏有数十个格子），`page_info` 是有限视图。若严格按原始 UI 顺序截断，大量无文本课程格可能挤掉左侧“第1周”等带文本导航锚点，影响即时决策。现已将**展示顺序**与**定位顺序**分离：

- `click(index=n)` 和所有工具输出的全局 `[n]` 始终取自当前屏幕可见的真实 clickable 序列，绝不重排；
- 混合页面中，`page_info` 最多展示 **60 个带文本 clickable + 8 个无文本代表项**（总计最多 68 项）；
- 每条展示项保留其真实全局 `[n]`，因此例如先展示 `[23] 第1周`、后展示 `[5] <无文本>` 不会改变任何点击语义；
- 仅有无文本项或仅有带文本项时，仍按原始顺序展示前 60 项；
- bounds 与屏幕矩形完全无交集的元素在 `SmartPerceiver._heuristic_understand()` 阶段被丢弃；边缘部分可见的元素保留。

当前屏幕内元素过多、超出 `page_info` 上限时，会提示调用：

```text
get_screen_info(mode="clickable", offset=...)
```

`get_screen_info(mode="clickable")` 支持 `offset` / `limit` 分页（单页 `limit` 最多 100），并在返回中说明下一页 offset。该分页仅用于浏览**当前屏幕内**的剩余可点击元素，而不是查看屏幕外内容。该模式只对**有 label** 的展示项按 label 去重；每一个无文本可点击项都保留自己的全局 `[n]`、bounds、class、path，绝不合并或丢弃。

### 5.3 面板摘要与告警

面板摘要现区分真实可点击数量和带文本数量：

```text
right_content bounds=[...] clickable_total=12 labeled_clickable=0 global_indexes=5,6,...
! 右侧存在 12 个无文本可点击元素；使用其全局 [n]，不要改用左侧导航的 [n]。
  示例：[5] <无文本> class=FrameLayout bounds=[...] path=...
```

只有右侧面板的 `clickable_total=0` 时，才输出“没有真实可点击元素”的告警。这样保留了阻止跨面板误点的约束，不再错误地劝 Agent 放弃右侧。

### 5.4 索引语义变更影响

这是有意的兼容性变更：无文本可点击元素会插入既有全局序号，任何硬编码 `click(index=...)` 的调用或测试都可能漂移。实施时已检索现有测试中依赖 index 的用例，并新增回归测试覆盖：

- 无文本 `list_entry` 的 `[n]` 可在当前屏幕可见的页面/工具输出中出现；
- `get_screen_info(mode="clickable", offset=...)` 可对当前屏幕内元素分页查看；
- `click(index=n)` 会直接命中当前屏幕内无文本可点击父节点；
- 原有带文本索引用例保持覆盖。

后续发布说明/团队通知必须注明：**索引语义已修正为全局 `clickable`，不再过滤无文本项。**

## 6. 已实施：P2 分栏防御约束

`_detect_pane_regions()` 仍从横向中心间隙推导候选分隔线，但已经将双栏分隔线限制为：

```python
width * 0.15 < split_x <= width * 0.45
```

当前双栏模型的语义是“左侧窄导航 + 右侧内容”；该约束阻止右侧多列内容内部的间隙被错误选为主分隔线。已增加回归测试，确认左侧宽度达到页面一半以上时不会声明 `left_navigation` / `right_content` 双栏。

该约束是防御性收紧，不取代后续的多列右栏样例和更系统的聚类/评分评审。
