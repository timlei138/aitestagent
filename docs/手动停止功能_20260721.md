# 手动停止运行功能 — 需求与实现方案

> 文档版本：2026-07-21
> 状态：待 Review
> 范围：Agent 运行中（同步 HTTP / SSE 流 / WebSocket 三入口）由用户主动停止当前任务

---

## 0. 背景与现状

### 0.1 用户场景

当前 Agent 跑长任务时（比如 `python main.py run "无限工作台添加计算器到桌面..."`），一旦模型进入"反复空转/撞歧义/卡死"的状态，用户**没有任何手段在不杀进程的前提下中止**——只能：

1. 关闭整个 webview / 杀掉后端进程（粗暴，影响其他正在跑的会话）
2. 等到 `max_turns` 耗尽（默认 ~35 轮，慢）
3. 等设备异常 / 网络异常触发 `exhausted` / `error` 路径（不可控）

> 在评审文档 `综合评审与问题分类_20260711.md §13` 的多轮真机复盘中，已多次出现「撞 max_turns → 几百秒后才发现」的低效循环——手动停止能力可大幅缩短恢复时间。

### 0.2 现状分析

| 关注点 | 现状 |
|---|---|
| **执行入口** | 三条：`POST /api/run`（同步）/`POST /api/run/stream`（SSE）/`/ws/chat` 的 `run` 消息 |
| **运行句柄** | `TestOrchestrator.start/start_stream` 直接调用 LangGraph `graph.invoke / graph.astream_events`，**没有 future / task 句柄暴露** |
| **thread_id 来源** | 三入口都**由后端自动生成**（`test-YYYYMMDD_HHMMSS`），前端**不感知** |
| **可中断点** | LangGraph `invoke/astream_events` 本身不提供外部 cancel API；只能在节点边界插桩 |
| **状态字段** | `TestState.execution_status` 枚举已知存在 `cancelled`（前端 `App.vue:562` 已定义映射），但**代码从未写入该值** |
| **可观测** | `orchestrator` 通过 `self._event_callback` + `set_ws_emit_callback` 把事件广播到 WebSocket，**`/api/run` 同步路径不广播** |
| **前端按钮** | 无任何「停止 / 取消」按钮；`WorkspacePanel.vue` 只有「执行」入口 |

### 0.3 设计原则（沿用评审文档 §0）

- **代码兜底「稳」，LLM 负责「活」**：停止是基础设施能力（用户意图 → 终止信号 → 状态收敛），属于「稳」的契约。
- **契约收敛，补丁发散**：停止信号用单一 `threading.Event` + 状态字段 `execution_status="cancelled"`，不引入多套并行机制。
- **不破坏现有语义**：补 `cancelled` 状态分支，不影响 `completed/exhausted/error/device_offline` 既有路径；reporter_node 的「全过归正 completed」逻辑（V1 修复）在 stop 时**不应触发**——停止本身就不算"通过"。
- **不在 click 指令进行中强行打断**：Android 设备上 click 过程中强杀可能留下半按状态；节点边界检查 + 工具内短暂让出，是更稳的收尾点。

## 1. 需求目标

### 1.1 功能性目标

| # | 用户故事 | 验收点 |
|---|---|---|
| F1 | 用户在 Agent 运行中点「停止」按钮，1 个节点周期内（≤ 5s）应停止继续推进 | 当前正在执行的工具调用返回后立即进入 `cancelled` 状态；不再发起新的 LLM/工具调用 |
| F2 | 停止后报告与数据库记录中应明确看到「用户手动停止」 | `execution_status="cancelled"`、`test_verdict="inconclusive"`、`conclusion` 含 `ABORT: USER_STOPPED` |
| F3 | 三条入口（HTTP 同步 / SSE 流 / WebSocket）均支持停止 | 每条入口都有对应的 stop 通道 |
| F4 | 停止操作对前端是「软停止」——按钮变 loading、最终收到 `run_stopped` 事件 | 不需要刷新页面 |
| F5 | 同一 thread 第二次点停止：幂等，无副作用 | 重复 stop 不抛异常、不影响其他会话 |
| F6 | 停止不影响其他正在跑的会话（如果未来支持多 run） | 通过 thread_id 隔离 |

### 1.2 非目标（Out of Scope）

- **不**实现「暂停后继续」（区别于 stop + 后续 resume；当前已有 plan_review 中断流，不需要重新设计）
- **不**实现「超时自动停止」（属于「最大步数/时间预算」范畴，已在 budget 机制内）
- **不**实现「远程/命令行停止」（仅前端按钮触发；CLI 路径 `main.py run` 仍由用户 Ctrl+C 中断进程）
- **不**改动 LangGraph 内部 `interrupt` 机制（plan_review_node 的现有中断流保持原状）

### 2.1 前端交互

```
└──────────────────────────────────────────────────────────────┘
```

### 2.2 后端时序（以 WebSocket 路径为例）

```
T+0.0s  WS recv: {type: "run", message: "..."}
T+0.0s  orchestrator.start(user_request) → 生成 thread_id=test-20260721_103000
T+0.0s  注册 _stop_flags[thread_id] = Event()
T+0.0s  yield/broadcast: {type: "run_started", thread_id: "test-..."}
T+0.1s  graph.invoke 开始 → planner_node → ... → agent_node
T+0.5s  agent_node 调用 LLM，等待响应中
T+8.0s  WS recv: {type: "stop_run", thread_id: "test-..."}
T+8.0s  orchestrator.request_stop(thread_id) → _stop_flags[thread_id].set()
T+8.5s  当前 LLM 响应到达；_run_agent 主循环迭代开头检查 stop → 命中
T+8.5s  agent_node 末尾 return Command(update={"conclusion": "ABORT: USER_STOPPED"})
         （或直接 raise RunStopped 让 graph 自然停止，路径下面讨论）
T+8.6s  graph 收敛到 reporter_node
T+8.7s  reporter_node 检查 stop → execution_status="cancelled" → test_verdict="inconclusive"
T+8.7s  广播 {type: "result", content: {execution_status: "cancelled", ...}}
T+8.7s  orchestrator.start 清理 _stop_flags，return
T+8.8s  WS send: {type: "run_stopped", thread_id: "test-..."}
```

### 2.3 异常分支

| 场景 | 行为 |
|---|---|
| 用户在 LLM 长时间生成中点停止 | 当前 LLM 响应继续接受，不 kill；下一轮节点入口检查到 stop → 立即中止 |
| 用户在 click 工具执行中点停止 | click 完成；下一轮节点入口检查到 stop → 立即中止（**不**中断正在执行的 click） |
| 用户在 plan_review interrupt 中点停止 | 停止当前 pending 的"等待人工"状态；等价于用户在 plan dialog 点"取消"——复用 `action=cancel` 路径 |
| 停止时 LLM 已返回、agent_node 正在写 state | 写完 state 后再让 graph 收敛到 reporter；reporter 看到 stop → cancelled |
| 同一 thread 停止信号到达时 run 早已结束 | 幂等：直接 return，清理 flag |
| 停止后用户立刻重启同一 thread_id | 不冲突：原 flag 清理后才允许新 run 注册同名 flag |

### 3.2 数据契约

**新增状态值**：

- `TestState.execution_status` 新增 `"cancelled"` 枚举值（前端 `execStatusMap` 已就绪）

**新增异常**：


**新增 WebSocket 消息类型**：

| 方向 | type | payload | 说明 |
|---|---|---|---|
| 后端→前端 | `run_started` | `{thread_id, started_at}` | run 启动时广播，前端用来记录 thread_id 并启用「停止」按钮 |

#### 3.4.3 `agents/orchestrator.py` — stop 入口与生命周期

```python
class TestOrchestrator:
    def __init__(self, config: TestConfig):
        # ... 既有 ...
        self._stop_flags: dict[str, threading.Event] = {}
        self._active_runs: dict[str, dict[str, Any]] = {}  # tid → {started_at, mode}

    def request_stop(self, thread_id: str, reason: str = "user_requested") -> bool:
        """幂等地为指定 thread 置位停止标志。返回 True 表示确实置位了新信号。"""
        ev = self._stop_flags.get(thread_id)
        if ev is None:
            logger.info("request_stop: no active run for thread %s (already done?)", thread_id)
            return False
        if not ev.is_set():
            ev.set()
            self._emit("status", {"type": "stopping", "thread_id": thread_id, "reason": reason})
            logger.info("request_stop: stop flag set for thread %s", thread_id)
        return True

    def _register_run(self, thread_id: str) -> threading.Event:
        ev = threading.Event()
        self._stop_flags[thread_id] = ev
        self._active_runs[thread_id] = {
            "started_at": datetime.now().isoformat(),
            "mode": "run",
        }
        return ev

    def _cleanup_run(self, thread_id: str) -> None:
        self._stop_flags.pop(thread_id, None)
        self._active_runs.pop(thread_id, None)

    def _attach_stop_event(self, ctx, ev: threading.Event) -> None:
        """把 Event 挂到 ToolContext 让节点检查。"""
        if ctx is not None:
            try:
                ctx._stop_event = ev
            except Exception:
                pass

    def start(self, user_request, app_package="", app_name="", thread_id="") -> dict:
        # ... 既有设备检查 ...
        if not thread_id:
            thread_id = f"test-{datetime.now().strftime('%Y%m%d_%H%M%S')}"

        # 注册 stop flag
        ev = self._register_run(thread_id)
        ctx = get_tool_context()
        self._attach_stop_event(ctx, ev)

        # 广播 run_started（让前端能拿到 thread_id）
        self._emit("run_started", {"thread_id": thread_id, "started_at": datetime.now().isoformat()})

        try:
            final_state = self.graph.invoke(initial_state, config_ctx)
            # ... 既有 interrupt 检测 ...
            return self._build_result(thread_id, final_state)
        except RunStopped:
            # graph 自然收敛到 reporter，reporter 写 cancelled
            final_state = self.graph.get_state(config_ctx)
            state = final_state.values if final_state else {}
            self._emit("run_stopped", {"thread_id": thread_id, "reason": "user_requested"})
            return self._build_result(thread_id, state)
        except Exception as exc:
            if "GraphInterrupt" in type(exc).__name__:
                # ... 既有 ...
                return self._handle_exception(thread_id, exc)
            return self._handle_exception(thread_id, exc)
        finally:
            self._cleanup_run(thread_id)
            # 清 ctx 上的 event 引用，避免下次 run 看到上一次的
            if ctx is not None:
                try:
                    ctx._stop_event = None
                except Exception:
                    pass
            run_log["cleanup"]()
```

#### 3.4.4 `agents/nodes.py` — 节点入口 stop 检查

```python
# agents/nodes.py 顶部

class RunStopped(Exception):
    """User-requested stop. 节点检查后抛出，graph 自然收敛。"""
    pass


def _check_stop(ctx) -> bool:
    """检查 stop 标志，命中时返回 True。绝不抛异常。"""
    if ctx is None:
        return False
    ev = getattr(ctx, "_stop_event", None)
    if ev is not None and ev.is_set():
        return True
    return False


# 在 agent_node / planner_node / reporter_node / plan_review_node 入口加：
def _stop_or_continue(state, ctx) -> Command | None:
    """命中 stop 时直接返回一个让图终止的 Command；不命中返回 None。"""
    if not _check_stop(ctx):
        return None
    return Command(
        update={
            "conclusion": "ABORT: USER_STOPPED — 用户手动停止当前运行",
            "status": "stopped",
            "_stop_requested": True,
        },
        goto="reporter_node",  # 强制收敛到 reporter
    )
```

具体接入位置：

- `agent_node` 入口（perceive 之前）：调用 `_stop_or_continue`，返回非 None 时直接 return。
- `planner_node` 入口（解析 user_request 之前）：同上。
- `reporter_node` 开头：检查 stop，命中时把 `execution_status` 强制写为 `"cancelled"`，`test_verdict="inconclusive"`，并在 `conclusion` 写 `ABORT: USER_STOPPED`。

#### 3.4.6 `api/server.py` — 新 endpoint + WS 分支

```python
class StopRunRequest(BaseModel):
    thread_id: str


@app.post("/api/run/stop")

    ok = orchestrator.request_stop(request.thread_id, reason="http_stop")
```

## 5. 风险与回滚

| 风险 | 缓解 |
|---|---|
| LLM 长时间生成中停止无效 | 文档已说明：当前 LLM 响应会接受完才停；接受 1 个 ≤ 5s 的尾巴。后续可加 LLM 端 cancel（langchain `RemoteRunnable` abort） |
| click 中强停导致半按 | 设计**不在**工具执行中断；节点边界检查 = 等 click 返回 = 安全 |
| 停止信号被吞（前端发 /api/run/stop 但后端未注册） | `request_stop` 返回 False → 前端 UI 不进入 stopped 态 → 用户重试 |
| reporter_node 改 cancel 判定破坏 V1 | stop 标志在 V1 判定**之前**拦截；V1 不动 |
| 与 plan_review 中断混淆 | stop 路径不调 `Command(resume=...)`；走自然的 graph 收敛 |
| 误清 ctx._stop_event 导致下个 run 看到上一次的 | 在 `start` finally 清回 None；新 run 注册新 Event |

**回滚**：本次改动只新增字段、新增 endpoint、新增按钮，不删任何已有逻辑。如需回滚：

1. 删除 `api/server.py` 的 `POST /api/run/stop` 与 WS `stop_run` 分支
2. 还原 `agents/orchestrator.py` 的 stop 注册/清理代码
3. 还原 `agents/nodes.py` 入口检查
4. 删除前端停止按钮

无 schema migration，无数据迁移。

## 7. 待 Review 决策点

1. **停止是否允许"取消正在执行 click"？** —— 方案选「不打断 click，节点边界检查」。备选：抛 `asyncio.CancelledError` 到工具层（复杂度高、对设备半按有风险）。**默认不打断**。
2. **停止后是否允许同一 thread_id 立即 resume？** —— 方案选「不允许；stop 是终点，resume 需新 thread」。备选：stop 视为另一种 interrupt，可 resume。**默认不允许**。
3. **CLI 路径 `python main.py run` 是否支持停止？** —— 方案选「不支持，仍由 Ctrl+C 中断进程」。备选：注册 SIGINT handler 调 request_stop。**默认不支持**（无 thread_id 感知）。
4. **`run_started` 是否要带回 `app_package` 等元信息？** —— 方案选「最小化：只带 thread_id + started_at」。备选：全量 echo。**默认最小化**。
5. **停止按钮是否要二次确认（confirm 对话框）？** —— 方案选「不二次确认，1 次点击即停」。备选：弹 1s 倒计时确认。**默认不确认**（用户场景就是"明显出错"）。
