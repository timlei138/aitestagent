# 用例计划质量提升方案评审

> 评审日期：2026-07-22  
> 评审对象：`docs/用例计划质量提升_execution_plan_20260722.md` v1  
> 依据：2026-07-22 三次 WLAN 执行 trace/log、当前数据库/API/Agent/工具实现  
> 结论：**方向正确，但不能按现稿直接实施；需先补齐执行证据、工具契约和 lineage 数据契约。**

---

## 1. 评审准绳

本评审遵循以下边界：

- **LLM 负责活**：理解当前页面、判断前置条件、选择下一步、处理页面漂移与语义歧义。
- **代码负责稳**：持久化可核验事实、统一工具返回码、校验输入、稳定等待、循环控制、终止契约和数据追溯。
- 允许增加有限且通用的契约、不变量和基础设施；不应把历史场景路径写成不断增长的 App 特例补丁。
- 复跑的目标应是“给 LLM 更强的成功证据”，而不是把 LLM 降级成盲目执行历史脚本的解释器。

---

## 2. 总体判断

`execution_plan` 的核心方向成立：当前复跑只复用了 `goal_json` 中的自然语言目标与 hints，未复用成功执行中的入口、定位和验证证据。把成功轨迹中的**结构化事实**沉淀下来，能减少 Agent 的重复探索，符合 Agent 能力演进方向。

但当前方案有三个 P0 阻塞项：

1. `test_runs.steps_json` 没有保存提炼器所需的真实工具参数与定位事实，无法可靠反向提炼 locator。
2. `EXECUTION_PLAN_BLOCK` 中“严格顺序执行、不要自行加步骤”的设计压缩了 LLM 的决策空间，也会在页面状态变化时带来误操作风险。
3. 105055 的高效率来自不同的初始状态，不能直接归因于计划复用或作为未来收益上限。

因此，建议先将方案重构为“**replay evidence（成功执行证据）注入**”，再逐步实现可执行计划能力。

## 3. 三次运行数据复核

| 指标 | 103153 普通运行 | 104626 复跑 #1 | 105055 复跑 #2 |
|---|---:|---:|---:|
| 时长 | 191.3s | 127.0s | 59.1s |
| LLM 调用 | 25 | 21 | 9 |
| 总 token | 631,319 | 360,910 | 98,669 |
| 步数 | 15 | 13 | 6 |
| 点击数 | 4 | 4 | 2 |
| verdict | passed | passed | passed |

### 已证实的结论

- 104626 跳过 planner 后，相比普通运行减少了 4 次 LLM 调用、约 43% 总 token 和约 34% 时长；复跑机制总体有效。
- 104626 在主页反复查找 WLAN，发生两次 `NOT_FOUND` 并误入“更多连接”，说明自然语言 hints 无法提供足够的导航事实。
- 105055 以 6 步完成了 WLAN 打开、客观断言、关闭、报告；两次目标点击均成功。

### 必须修正的归因

105055 的起始页面已经是 `SubSettings` 的 WLAN 设置页，页面上直接存在 `[3] WLAN [switch_row]`。它不需要从 Launcher/Settings 首页导航；同时上下文已有相同 App 的 RAG 操作经验。因此其 9 次 LLM、6 步、59 秒不能单独证明“复用计划接近上限”。

后续 A/B 测试必须统一：设备状态、App 版本、初始 Activity、WLAN 初始状态和网络条件；在相同条件下分别运行普通模式、旧复跑和携带 replay evidence 的复跑。

## 4. P0：执行证据数据契约缺失

方案假设 `test_runs.steps_json` 能够读取如下字段：

```json
{
  "action_type": "click",
  "tool_input": {
    "label": "WLAN",
    "index": 3,
    "class_name": "LinearLayout",
    "path_contains": "..."
  }
}
```

但当前 `agents/orchestrator.py::_build_display_steps()` 实际只落库 `action_type`、`target`、`observation`、`status` 等展示字段；没有保存 `tool_input`、`rid`、`index`、`class_name`、`path_contains`、`role`、页面前后签名或结构化状态码。

`logs/runs/*_trace.json` 虽然带有部分 `tool_input`，但 API 提炼器读取的是数据库的 `test_runs.steps_json`，不能把临时 trace/log 当成生产数据源。按现状实现 `_extract_execution_plan()`，会提炼出空 locator，或只能从 observation 文本猜测事实。

### 建议：先记录版本化执行事件

每次工具调用应落成通用、可追溯的结构化事件，例如：

```json
{
  "schema_version": 1,
  "tool": "click",
  "args": {
    "label": "WLAN",
    "rid": "",
    "class_name": "LinearLayout",
    "path_contains": "content_frame > main_content",
    "index": 3
  },
  "status_code": "OK",
  "match_mode": "exact",
  "fallback_used": false,
  "page_before": {"activity": "Settings$WifiSettingsActivity", "signature": "..."},
  "resolved_target": {
    "label": "WLAN",
    "rid": "android:id/switch_widget",
    "class_name": "LinearLayout",
    "role": "switch_row",
    "path": "..."
  },
  "page_after": {"activity": "Settings$WifiSettingsActivity", "signature": "..."}
}
```

这属于运行基础设施：代码只记录客观事实，不替 LLM 决定“下一步应该点什么”。

## 5. P0：不要把 replay evidence 变成强制脚本

以下规则不建议进入 prompt：

- “严格按顺序执行”；
- “不要自行加步骤”；
- “不应自己插入 launch_app / scroll_find_and_click / press_key”。

历史成功路径只在当时的页面、设备、网络、应用版本和弹窗状态下有效。当前状态不满足历史前置条件时，强制执行会使 LLM 使用过期 index 或错误入口。

建议将 `EXECUTION_PLAN_BLOCK` 改为 **`REPLAY_EVIDENCE_BLOCK`**：

> 该块提供历史成功运行中的高置信入口、目标元素事实、定位偏好和验证证据。当前页面满足前置条件时优先采用；不满足时先重新感知，再由 Agent 判断恢复策略。禁止盲用历史 index。

LLM 保持以下职责：判断当前页是否匹配、决定是否启动入口、判断是否采用证据路径、发生偏差时选择回退策略。代码提供事实、校验和错误码。

## 6. P0：schema 必须与真实工具接口一致

### 6.1 Activity 入口

当前 Agent 注册的是：

```python
launch_app(package: str, activity: str = "")
```

并不存在 `launch_activity` 工具。计划 schema 可以使用 `entry.kind="activity"` 作为描述，但实际渲染给 Agent 必须是：

```text
launch_app(package="com.android.settings", activity="com.android.settings.Settings$WifiSettingsActivity")
```

入口工具还应返回启动后的实际 Activity/页面事实；仅返回“已启动”不足以确认请求的入口成功到达。

### 6.2 role 与 click 参数

当前 `click()` 支持 `label`、`alternatives`、`rid`、`class_name`、`path_contains`、`index`，**不支持 `role`**。因此 `role="switch_row"` 应存为预期目标事实，不能渲染成 `click()` 参数。

### 6.3 index 不是跨运行稳定 locator

当前工具只要收到 `index`，就直接按当前页面可点击元素的全局位置选择，且不再用 `rid/class_name/path_contains/label` 做联合过滤。页面列表、动态网络项、弹窗和布局变化都可能使同一个 index 指向不同元素。

推荐 locator 契约：

1. 优先 `rid + label`；
2. 其次 `class_name + path_contains + label`；
3. `index` 仅作为最新一次页面感知下的临时消歧手段；
4. 使用 index 前校验当前 Activity、页面签名和预期目标事实；
5. 不匹配则返回 `PRECONDITION_NOT_MET`/`NOT_FOUND`，交由 LLM 重新感知和决策。

## 7. P1：验证证据不能只保留 assert_verification

`assert_verification` 是 Agent 的结构化报告，不是独立的 ground truth。它不能单独证明 WLAN 已开启或关闭。

105055 中 v0 的强证据是：

```text
assert_page_contains("已连接") -> PASS
```

如果新计划只保留 `click + assert_verification + report_done`，反而会丢失已经成功使用过的客观断言。

建议为每个 verification key 保存证据约束：

- `assert_page_contains`；
- `assert_element_exists`；
- 后续可增加通用的 `assert_element_state(locator, expected_state)`；
- 无法客观验证的结论必须保留 `unknown/review_required`，不能为了减少步数直接报 passed。

对于 WLAN，开关的可感知 `on/off` 状态应是主证据；“已连接”等页面文字可作为辅助证据。

---

## 8. P1：当前提炼算法会污染成功计划

现稿按动作类型筛选 `click/assert_verification/report_done` 存在风险：

- 目前展示步骤的 `status` 大多为 `continue`，不能区分 `OK`、`NOT_FOUND` 和无效导航；
- 103153 中失败的精确点击也可能被提炼为“关键动作”；
- 取第一个 `launch_app` 不能保证是与验证动作相关的正确入口；103153 后续才进入 WLAN Activity；
- 按 `assert_verification` 出现顺序绑定此前 click，会把导航、失败点击和业务点击混入同一个验证项；
- 从 observation 字符串正则解析 Activity 脆弱，应直接持久化真实工具参数与启动后页面事实。

提炼器应只从 `completed + passed + 证据完整` 的运行中提取，并按 verification 结果分割成功片段。无法可靠关联的步骤应不进入 replay evidence；宁可让 LLM 正常探索，也不要把失败历史伪装成确定契约。

---

## 9. P1：lineage 应类型化

当前 `test_runs.source_run_id` 有两种语义：

- 从报告直接复跑时保存 report run ID；
- 从用例运行时保存 case ID。

因此 105055 的 `581d1532b90e` 很可能是 case ID，不应直接判定为“脏 run_id”。真正问题是同一字段混存不同实体 ID，导致追溯无法可靠查询。

建议至少拆分为：

```text
source_case_id: nullable
source_run_id: nullable
execution_plan_revision: integer
```

用例运行应同时记录来源 case 和该 case 原始的 source run。来源报告被删除时，UI 可以提示“原报告不可用”，但不应影响已内嵌执行证据的用例继续运行。

---

## 10. P2：plan_quality_json 不宜单独落库

`plan_quality_json` 与 `goal_json.execution_plan` 会构成两份可变事实源。用户编辑 execution plan 后，质量字段可能没有同步更新，产生陈旧 UI 信息。

建议：

- 仅持久化 `goal_json.execution_plan`；
- API 读取用例时即时计算 `plan_quality` 并返回；
- UI 展示派生结果但不将其作为独立真相保存。

“定位字段填充数”也不能单独作为质量指标：不稳定的 index 与稳定的 resource id 在计数上权重相同。更有效的质量信号包括：入口是否可验证、是否有稳定 locator、是否存在页面前置条件、verification 是否绑定客观断言、来源运行是否通过、最近一次受控复跑是否成功。

---

## 11. Token 与步骤优化建议

三次运行都具有较高缓存命中率，应把 raw token 与新增上下文成本分开观察：

| Run | 输入 token | 缓存输入 | 未缓存输入 | 输出 token |
|---|---:|---:|---:|---:|
| 103153 | 627,639 | 597,760 | 29,879 | 3,680 |
| 104626 | 358,166 | 341,120 | 17,046 | 2,744 |
| 105055 | 96,863 | 90,112 | 6,751 | 1,806 |

当前每轮会累积 Goal、Page、History、RAG 与工具对话。建议建立通用的上下文压缩契约：

- 永远保留系统契约、Goal、最新页面事实、replay evidence、验证状态；
- 仅保留最近 N 条工具调用与结构化结果；
- 已完成验证压缩成结构化摘要，不重复携带历史 UI tree；
- RAG 仅保留当前 App/页面相关且未被 execution evidence 覆盖的片段；
- 同时观测 `cached_input_tokens`、`uncached_input_tokens`、输出 token、LLM 调用、端到端时长和客观验证覆盖率。

这是通用上下文基础设施优化，不是针对 WLAN 的补丁。

---

## 12. 点击质量指标应修正

“exact”只表示调用带有精确参数，不表示操作成功或目标正确。103153 的 `exact_click_count=4` 中仍有一次 `NOT_FOUND`；104626 是 3 次 exact、1 次 semantic，且 `fuzzy_click_count=0`。

建议将指标改为：

- `successful_exact_click_count`；
- `failed_exact_click_count`；
- `successful_semantic_click_count`；
- `ambiguous_count`；
- `fallback_used_count`；
- `verification_evidence_coverage`。

优化目标应是“被后置页面事实或断言支持的成功操作率”，不是简单追求所有 click 都使用 exact 参数。

---

## 13. 推荐实施顺序

1. **执行证据落库**：持久化真实 tool args、状态码、目标解析事实和页面前后签名。
2. **lineage 数据契约**：拆分 `source_case_id` 与 `source_run_id`，记录 plan revision。
3. **replay evidence 提炼器**：仅从证据完整的成功运行中提炼；失败或不确定时不生成确定路径。
4. **实际 prompt 注入点改造**：当前 `agent_node` 以初始 `SystemMessage(AGENT_SYSTEM)` 加动态 HumanMessage 组织上下文，并不存在方案假设的 `_render_system_prompt()`；应基于真实调用链加入 evidence block。
5. **入口与 locator 通用契约**：校验入口是否真正到达、禁止跨页面盲用 index、保留 LLM 回退决策权。
6. **客观验证证据**：在 replay evidence 中保留可复用断言，而不是只保留 Agent 报告。
7. **UI 编辑与质量标签**：后端 schema 校验、工具白名单和版本校验优先于自由编辑。
8. **受控 A/B 测试**：统一起始状态后，分别测试普通运行、旧复跑和 replay-evidence 复跑。

---

## 14. 最终决策建议

| 项目 | 结论 |
|---|---|
| 从成功轨迹提炼结构化复用信息 | ✅ 应做 |
| 先持久化工具与页面事实 | ✅ 必须先做 |
| 把历史轨迹变为强制脚本 | ❌ 不建议 |
| 强制“不许 LLM 自行增加步骤” | ❌ 不建议 |
| 用 replay evidence 帮助 LLM 决策 | ✅ 推荐 |
| 保存可复用的客观断言 | ✅ 推荐 |
| `source_run_id` 同时存 run/case | ❌ 必须拆分 |
| 单独持久化 plan_quality_json | ⚠️ 建议改为 API 派生字段 |
| 用 105055 推导未来性能目标 | ❌ 需先完成受控 A/B 验证 |

**一句话结论：代码应沉淀和校验“历史执行中已经证明的事实”，LLM 应依据这些事实理解当前页面并决定是否、如何复用路径；不要让代码或 prompt 将 Agent 固化为按历史脚本盲走。**
