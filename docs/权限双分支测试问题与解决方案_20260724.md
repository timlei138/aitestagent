# 权限双分支测试缺失：现象、根因与解决方案

> 指导原则（综合评审与问题分类_20260711.md §0）：  
> **代码负责"稳"（契约、事实、基础设施），LLM 负责"活"（理解、决策、判断）。**  
> 凡是让"稳"的部分收敛、让 LLM 更放得开的兜底，就做；凡是让代码陷入无限特例补丁的，就停下来换思路。

---

## 现象

测试用例要求验证相机权限和图库权限的**两种结果**：

| 触发操作 | 允许路径 | 拒绝路径 |
|----------|---------|----------|
| 点击"拍照导入课程表" | 选择允许后相机打开 | 选择拒绝后提示需要相机权限 |
| 点击"从图库导入课程表" | 选择允许/部分照片后可选照片 | 选择拒绝后提示需要相册权限 |

实际执行（run `test-20260724_111559`）只完成了两个拒绝路径，允许路径完全未覆盖，任务却被判定为 `done / passed`。

---

## 根因链

```
用户需求（4项）
      ↓
  [Planner]  ← 不识别"允许/拒绝"为独立分支，合并为2项（只含拒绝）
      ↓
goal.verification = [v0: 拒绝相机, v1: 拒绝图库]   ← 允许路径在规划阶段丢失
      ↓
  [Execution Plan]  ← 继承上述2项，key_actions 只覆盖拒绝流程
      ↓
  [Agent]  ← 无"分支回溯"概念；测完 v0、v1 直接 report_done
      ↓
  结果：2/2 passed（允许路径从未出现在 verification 列表中）
```

**三个独立失效点：**

1. **Planner 合并分支**：`## 多场景` 规则只认 `；;` 和 `同时` 作为分隔符，"允许/拒绝"写在同一条描述内被合并，"允许"分支在规划阶段就被丢弃。

2. **Agent 无双分支意识**：agent.txt 对权限弹窗的说明只涉及"如何点击"和"弹窗消失时如何处理"，没有引导 Agent 识别"verification 含两种对立结果，需要分轮执行"。

3. **Planner 不知道权限工具**：planner.txt 的 `## Agent 可用工具` 缺少 `set_runtime_permission`，无法生成 ADB 授权的 hints。

---

## 深层根因：LLM 响应时间 vs 权限弹窗超时

即使上述三个问题都修复，仍存在**结构性竞态**：

```
点击"拍照导入" → 权限弹窗弹出（系统级，约 10s 自动消失）
      ↓
  调用 wait_for_permission_dialog() → 拿到弹窗按钮列表
      ↓
  LLM 推理 + API 调用（5~15s）  ← 可能超过弹窗 10s 超时
      ↓
  调用 respond_to_permission_dialog(button="允许") → NOT_FOUND，弹窗已消失
```

LLM 的响应时间（含网络延迟）不可控，权限弹窗超时是系统硬编码，无法通过 adb 延长（已验证 ZUI/MIUI/ColorOS 均无此配置项）。这是**架构层问题**，不是 prompt 能解决的。

## 运行时发现（2026-07-24 14:13 run 日志分析）

三层架构代码已全部实施并测试通过（98 passed），但首次真实设备运行暴露了**两个 Plan 阶段未预见的问题**：

### 发现 1：中间介绍弹窗阻断第一层检测

实际执行链（以「拍照导入 deny」为例）：

```
click("拍照导入课程表", permission_hint="deny")   ← t=0
  └ _detect_permission_popup(timeout=2.0) 轮询 0~2s
  └ 期间 UI 上是 App 介绍弹窗（"请确保图片清晰、完整" + "知道了"）
  └ 不是 GrantPermissionsActivity → 全部返回 None
  └ t=2s：helper 返回 None → click 无 permission 字段
click("知道了")                                     ← t≈5s
  └ App 这才调 requestPermissions()
  └ GrantPermissionsActivity 出现                    ← t≈5.5s（已超 Layer 1 窗口 3s+）
```

**根因**：`permission_hint` 是预测式设计——Planner 赌哪一步出权限弹窗，但实际可能隔着 N 个中间弹窗。2s 轮询窗口无法穿透中间弹窗链。

### 发现 2：Layer 2 延迟 > 弹窗 10s 寿命

`respond_to_permission_dialog` 在弹窗明明存在时返回 `NOT_FOUND`（全轮 4 次权限交互全部如此）。原因：Layer 2 是两轮 LLM 往返（`wait_for_permission_dialog` 检测 → LLM 决策 → `respond_to_permission_dialog` 点击），LLM 决策延迟 + 网络延迟 > 弹窗 10s 自动消失。最终全靠 Layer 3（`set_runtime_permission` ADB 兜底）完成授权。

### 结论：预测式 `permission_hint` 模型有结构性缺陷

| 问题 | 根因 |
|------|------|
| 中间弹窗阻断检测 | Planner 无法预测中间弹窗链的存在和长度 |
| Layer 2 延迟竞态 | LLM 往返时间 > 弹窗寿命，结构性不可解 |
| Layer 1 从未真正生效 | 预测式 hint 在真实 App 流程中命中率极低 |

## 解决方案：三层权限处理架构

### 整体思路

```
第一层（主路径）：set_permission_intent(action) + click() — Reactive Intent 持续监听
       ↓ 匹配失败 / intent 未设 / 异常
第二层（降级）：respond_to_permission_dialog — LLM 手动处理弹窗
       ↓ 弹窗已超时消失
第三层（兜底）：set_runtime_permission — ADB 直接授权/撤销
```

### 第一层：Reactive Permission Intent — 声明意图 + 持续监听

#### 设计哲学

从**预测式**（Planner 赌哪一步出弹窗）改为**反应式**（声明测试意图，后续每次 click 自动监听）：

```
人测试的逻辑："我知道这轮要测 Camera 拒绝，
所以不管我点了什么，只要 Camera 权限弹窗出来了，我就点拒绝。"
```

不预测哪一步出弹窗、不特化处理中间弹窗——权限弹窗在哪次 click 里冒出来，就在哪次被处理。

#### 使用方式

```python
# 1. Agent 在测试分支开头声明意图
set_permission_intent(action="deny")         # 这轮测拒绝

# 2. 后续每次 click() 自动监听权限弹窗
click("拍照导入课程表")                        # 没看到权限弹窗，正常返回
click("知道了")                                # 介绍弹窗关掉 → 权限弹窗出现
                                               # → click 内部检测到 → 自动点"拒绝"
                                               # → evidence: permission_auto_handled=deny

# 3. 分支结束时清除
set_permission_intent()                        # 传空 = 清除
```

#### 与旧 `permission_hint` 的对比

| | 旧 permission_hint | 新 Reactive Intent |
|--|--|--|
| 决策点 | Planner 预测某一步 | Agent 声明测试目标 |
| 生效范围 | 仅那一次 click | 声明后所有 click 持续生效 |
| 中间弹窗 | 挡住 → 失败 | 不挡住 → 后续 click 继续监听 |
| 符合人的逻辑 | 不符合（预知未来） | 符合（看到才反应） |

#### 执行流程

```
set_permission_intent(action="deny")
  → ctx._permission_intent = {"permission": "camera", "action": "deny", "set_time": <monotonic>}

后续每次 click(label) 内部：
  ├─ 正常执行点击
  ├─ 检查 ctx._permission_intent 是否有效（非空 + 未过期）
  ├─ 有界轮询 600ms 检测权限弹窗
  │   ├─ 检测到 GrantPermissionsActivity → auto-click "拒绝" → evidence 回写
  │   └─ 未检测到 → 正常返回（中间弹窗不影响）
  └─ permission_hint 参数优先级仍高于 intent（向后兼容）
```

#### 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 轮询时长 | intent 模式 600ms，hint 保留 2s | intent 不预测时机，只检查"此刻有没有"；多次 click 自然累计 |
| 生命周期 | Agent 显式 clear + TTL 120s | 不做 fire-and-clear（某些 App 连续弹多个权限弹窗）；TTL 防 LLM 遗忘 |
| 权限类型匹配 | v1 不按类型匹配 | 按钮文案跟权限类型无关；activity 闸门已限定只处理权限弹窗；permission 字段保留为可选日志 |
| permission_hint 迁移 | 保留为 deprecated 薄壳 | 优先级高于 intent，内部走 2s one-shot 路径不改；老 hints 不失效 |

#### 证据回写

当 `click("知道了")` 内部顺手处理了权限弹窗时，evidence 包含：
```python
{
    "permission_dialog": "true",
    "permission_activity": ".permission.ui.GrantPermissionsActivity",
    "permission_buttons": "拒绝|允许",
    "permission_state": "awaiting_response",
    # 新增：自动处理标记
    "permission_auto_handled": "deny",          # 已自动点击"拒绝"
    "permission_auto_button": "拒绝",            # 实际点击的按钮文本
    "permission_auto_result": "handled",         # 处理结果
    "permission_auto_trigger": "知道了",          # 触发自动处理的 click label
}
```

`permission_auto_trigger` 让 Agent 理解"我点的是知道了，但权限弹窗也被顺手处理了"。

#### `permission_hint` 向后兼容

`permission_hint` 参数保留在 `click()` 签名上，优先级高于 `_permission_intent`：
```
if permission_hint:    → 2s one-shot 路径（现有逻辑不改）
elif intent 有效:      → 600ms 监听路径（新增）
else:                  → 非 hint 路径（现有逻辑不改，3 字段布尔值）
```

### 第二层：`respond_to_permission_dialog`（LLM 手动处理）

当第一层不覆盖时（hint 未传、按钮不匹配、auto-click 异常），LLM 通过 `wait_for_permission_dialog` + `respond_to_permission_dialog` 手动处理弹窗。这两个工具**保留不删除**，作为降级路径。

需要第二层的场景：

| 场景 | 原因 |
|------|------|
| Planner 未预见弹窗（没传 `permission_hint`） | 第一层未激活 |
| 按钮不匹配（`settings_required`、媒体部分访问等） | 第一层故意跳过 |
| auto-click 抛异常（UI 已消失） | 第一层 try/except 降级 |

### 第三层：`set_runtime_permission`（ADB 兜底）

弹窗已超时消失或需要分支间权限状态切换时，使用 ADB 直接授权/撤销：

| 场景 | 用法 |
|------|------|
| 弹窗超时消失 | `set_runtime_permission(grant/deny)`（注：`pm grant` 不触发 `onRequestPermissionsResult` 回调，需导航回触发点重新点击） |
| 分支间权限状态切换 | `set_runtime_permission(deny)` 撤销权限，让弹窗下次再弹出 |

#### perceive_tools.py：新增 `set_permission_intent` 工具

```python
@tool
def set_permission_intent(permission: str = "", action: str = "") -> str:
    """声明本轮权限测试意图。设置后，后续每次 click() 自动监听
    匹配的权限弹窗并按 action 响应，无需在 click 中传 permission_hint。

    permission: 可选权限类型（camera/location/storage 等），仅作日志标记
    action: "grant" 或 "deny"
    传空值清除意图。
    """
    ctx = get_tool_context()
    if not permission and not action:
        ctx._permission_intent = {}
        return make_result(OK, "权限测试意图已清除")
    if action not in ("grant", "deny"):
        return make_result(ERROR, "action 必须是 grant 或 deny")
    import time
    ctx._permission_intent = {
        "permission": permission.lower().strip(),
        "action": action.strip(),
        "set_time": time.monotonic(),
    }
    return make_result(OK,
        f"已设置权限测试意图: permission={permission}, action={action}。"
        f"后续 click() 将自动监听权限弹窗并按 {action} 响应。"
        f"测试完当前分支后调用 set_permission_intent() 清除。",
        {"permission": permission, "action": action},
    )
```

#### click.py：`_maybe_auto_handle_permission` 增加 intent 路径

```python
def _maybe_auto_handle_permission(ctx, permission_hint: str, label: str) -> dict | None:
    """权限弹窗自动处理 helper。三路分流：hint > intent > 默认。"""
    from tools.perceive_tools import (
        _detect_permission_popup, _match_permission_button,
        _permission_evidence, _permission_popup_buttons,
    )
    import time

    # ── 检查 intent 有效性（TTL 120s）──
    intent = getattr(ctx, "_permission_intent", None) or {}
    intent_valid = bool(intent) and (
        time.monotonic() - intent.get("set_time", 0) < 120.0
    )

    if permission_hint:
        # ── hint 路径：2s one-shot（现有逻辑不改）──
        info = _detect_permission_popup(ctx, timeout=2.0)
        if not info:
            return None
        activity, controls = info
        matched = _match_permission_button(controls, permission_hint)
        if not matched:
            return {**_permission_evidence(activity, controls),
                    "permission_auto_result": "fallback_match"}
        text, bounds = matched
        try:
            ctx.device.click_bounds(bounds)
            logger.info("click: auto-click permission | hint=%s button=%s", permission_hint, text)
            return {**_permission_evidence(activity, controls),
                    "permission_auto_handled": permission_hint,
                    "permission_auto_button": text,
                    "permission_auto_result": "handled",
                    "permission_auto_trigger": label}
        except Exception:
            logger.warning("click: auto-click permission failed | hint=%s", permission_hint)
            return {**_permission_evidence(activity, controls),
                    "permission_auto_result": "fallback_error"}

    elif intent_valid:
        # ── intent 路径：600ms 监听（新增）──
        info = _detect_permission_popup(ctx, timeout=0.6)
        if not info:
            return None
        activity, controls = info
        action = intent["action"]
        matched = _match_permission_button(controls, action)
        if not matched:
            return {**_permission_evidence(activity, controls),
                    "permission_auto_result": "fallback_match"}
        text, bounds = matched
        try:
            ctx.device.click_bounds(bounds)
            logger.info(
                "click: auto-click permission | intent=%s button=%s trigger=%s",
                action, text, label,
            )
            return {**_permission_evidence(activity, controls),
                    "permission_auto_handled": action,
                    "permission_auto_button": text,
                    "permission_auto_result": "handled",
                    "permission_auto_trigger": label,
                    "permission_intent_type": intent.get("permission", "")}
        except Exception:
            logger.warning("click: auto-click permission failed | intent=%s", action)
            return {**_permission_evidence(activity, controls),
                    "permission_auto_result": "fallback_error"}

    # ── 默认路径：保持改动前行为 ──
    info = _permission_popup_buttons(ctx)
    if info:
        activity, controls = info
        return {
            "permission_dialog": True,
            "permission_activity": activity,
            "permission_buttons": "|".join(text for text, _ in controls),
        }
    return None
```

## 实现约束（5 条，必须守住）

### 约束 1：窄口径——只处理权限弹窗

auto-click 只针对 `_PERMISSION_ACTIVITY_MARKERS`（`permissioncontroller`/`grantpermissionsactivity`）匹配的 Activity。业务提示框（如"请确保图片清晰"）不含这两个 marker，不会误触。

**绝不扩展到处理其他类型弹窗**（保存草稿、业务提示等）——一旦想加，就是"无限特例补丁"的苗头，应停下。

### 约束 2：click 必须回写"已自动处理"证据

click 内部自动点了按钮后，LLM 下一回合只拿到 click 的返回值。如果 evidence 不显式写明"已自动处理"，LLM 会失去知情权，可能重复调 `wait_for_permission_dialog`。

必须在 evidence 里追加（是合并，不替换现有 `_permission_evidence` 的 4 个字段）：
```python
# 现有字段保留
"permission_dialog": "true",
"permission_activity": activity,
"permission_buttons": "拒绝|允许",
"permission_state": "awaiting_response",
# 新增字段
"permission_auto_handled": "deny",     # 已自动点击"拒绝"
"permission_auto_button": "拒绝",       # 实际点击的按钮文本
"permission_auto_trigger": "知道了",    # 触发自动处理的 click label
```

### 约束 3：按钮匹配必须是显式确定性规则

```python
_GRANT_BUTTONS = ("仅在使用中允许", "仅本次使用时允许", "始终允许", "允许", "Allow")
_DENY_BUTTONS = ("拒绝", "不允许", "Deny", "Don't allow")
```

- 元组顺序 = 优先级（下标越小越优先，最小权限优先）
- 匹配逻辑：遍历 controls，在所有命中项中选 `_GRANT_BUTTONS`/`_DENY_BUTTONS` **下标最小**的那个（不是 controls 顺序第一个）
- **匹配不到 → 跳过**，交给 LLM
- `"前往设置"` / `"Go to settings"` 必须**显式排除**，不加进任何一侧（实现时加注释防误加）
- **匹配前对按钮文本做 `.strip()`**，防止首尾空格导致精确匹配失败
- **媒体「部分访问」弹窗故意不匹配**：Android 13/14 的 `READ_MEDIA_IMAGES` 权限弹窗按钮是「拒绝 | 只允许访问所选照片 | 允许访问所有照片」——"选哪种访问范围"是决策（归 LLM），不是确定性事实，匹配失败后自动落回第二层

### 约束 4：planner.txt 和 agent.txt 必须同步改

当前 planner.txt 仍在生成旧流程的 hints。如果只改 agent.txt 不改 planner.txt，Planner 产出的 hints 不会写 `click(permission_hint=...)`，Agent 接不住。**两个文件必须一起改**。

### 约束 5：`permission_hint` 保留为 deprecated 薄壳

`permission_hint` 参数保留在 `click()` 签名上，优先级高于 `_permission_intent`，内部走 2s one-shot 路径不改。新分支应使用 `set_permission_intent(action)` 替代。老 hints 中的 `permission_hint` 仍有效。

### 约束 6：intent 必须有 TTL 兜底

`_permission_intent` 设置时记录 `set_time`（`time.monotonic()`），click 内部检查时若超过 120s 则视为已过期、等同未设置。防止 LLM 忘记调 `set_permission_intent()` 清除导致残留 intent 在后续分支误触发。

## 实现补充（6 条）

### 补充 1：Photo Picker ≠ 权限弹窗

Android 13+ 的 `READ_MEDIA_IMAGES` 运行时权限弹窗 = `GrantPermissionsActivity`。

- **简单二元媒体弹窗**（按钮只有「拒绝/允许」）→ `_GRANT_BUTTONS` 匹配「允许」，第一层覆盖
- **带范围选项的媒体弹窗**（按钮是「拒绝/只允许访问所选照片/允许访问所有照片」）→ 这两个 grant 变体不在 `_GRANT_BUTTONS` 里，第一层故意不覆盖，落回第二层由 LLM 根据测试意图选择（见约束 3）

Android 13 的系统相册选择器（Photo Picker，`ACTION_PICK`）是另一个 Activity（如 `PhotosPickerActivity`），不含 `_PERMISSION_ACTIVITY_MARKERS` → **三层架构均不覆盖**，仍需普通 click/选择逻辑。

### 补充 2：弹窗检测必须用有界轮询，不是单次

`_permission_popup_buttons()` 是纯同步单次检测（`current_app` + `dump_hierarchy` 立即返回）。但 Android 在 `click_bounds` 后，`GrantPermissionsActivity` 有 ~100-300ms 的转场动画才把按钮渲染进层级树。如果 B 在点击后立刻只调一次检测，大概率拿到"无弹窗"→ 第一层落空，又变回 LLM 等待（竞态复发）。

必须用有界轮询（`_detect_permission_popup(ctx, timeout=2.0)`，每 200ms 检测一次），上限 2s 仍远小于弹窗 10s 超时，但能真实捕获弹窗。这个 helper 同时供 B 和 `wait_for_permission_dialog` 复用（DRY）。

### 补充 3：auto-click 自身失败要 try/except 降级

按钮匹配到了、但 `click_bounds` 抛异常（如 UI 已消失、坐标越界）——不能让 click() 因此 raise 中断整个点击动作（代码已在待实施变更的 helper 中体现）。

### 补充 4：ADB 兜底仍不触发回调

`set_runtime_permission(grant)` 本质是 `pm grant`，不触发 `onRequestPermissionsResult` 回调。使用 ADB 兜底后，仍需**导航回触发点并重新点击**，让 App 再次调 `requestPermissions()`，此时 Android 发现权限已 GRANTED，直接触发回调。

### 补充 5：auto-click 加日志打点

B 的成功/降级应在日志中明确记录（和"LLM 决策执行"区分）：
```python
logger.info("click: auto-click permission | hint=%s button=%s", hint, matched_text)
logger.warning("click: auto-click permission failed | hint=%s", hint)
```
便于事后统计"第一层触发次数 / 匹配成功率 / 降级到第二层的比率"，验证竞态是否真正消除。

在 evidence 中追加 `permission_auto_result` 字段便于量化聚合：
- `"handled"`：B 自动点选成功
- `"fallback_match"`：按钮不匹配，落回第二层
- `"fallback_error"`：点选抛异常，落回第二层

### 补充 6：Reactive Intent 解决中间弹窗问题

真实 App 流程中，触发权限请求前可能弹出 N 个介绍/引导弹窗（如"请确保图片清晰"）。这些弹窗不是 `GrantPermissionsActivity`，activity 闸门不会误触。Reactive Intent 模式下，中间弹窗不影响监听——权限弹窗在哪次 click 里出现，就在哪次被处理，无需预测、无需特判。

## 验证方法

重新执行同一测试用例，检查：
1. Planner 生成的 `verification` 含 4 项（允许+拒绝 × 相机+图库）
2. Planner hints 包含 `set_permission_intent(action=...)` + `click(触发操作)`（不传 permission_hint）
3. Agent 在每个分支开头调 `set_permission_intent`，结尾调 `set_permission_intent()` 清除
4. 中间介绍弹窗（"知道了"/"请确保图片清晰"）不阻断权限检测——权限弹窗在后续 click 中被自动处理
5. click 返回的 evidence 包含 `permission_auto_handled` + `permission_auto_trigger` 字段
6. 按钮不匹配或弹窗未出现时，正确降级到 `respond_to_permission_dialog` 或 `set_runtime_permission`
7. 当 intent 未设置时，click 行为与修改前完全一致（不影响现有非权限场景）
8. TTL 120s 后 intent 自动过期，不会在后续分支误触发
