## 0) 指导原则：给 Agent 必要的代码兜底，让 LLM 发挥更大作用

Agent 进化的方向不是「用代码替 LLM 做事」，而是**用代码搭好稳固的地基（契约 / 事实 / 基础设施），让 LLM 在其上放开手脚做它真正擅长的事**（理解页面、找元素、决策下一步、语义判断）。下面所有建议都以此为准绳。

**边界怎么划——「判断 vs 事实/契约」：**

| 类型 | 例子 | 归属 |
|---|---|---|
| **感知 / 决策 / 语义判断**（有歧义、要理解） | 该点哪个元素、下一步做什么、"购物车是否显示了刚加的商品" | **交给 LLM**（代码正在从这里**退出**，如 legacy 元素匹配下线） |
| **确定性事实**（可被程序核实） | 任务是否已调用 report_done、页面**字面**是否含"已支付"、元素是否存在 | 代码给 **ground truth**，别让 LLM 靠"看一眼"来猜 |
| **工具契约 / 运行基础设施**（有限且稳定） | click 统一返回 OK/NOT_FOUND/AMBIGUOUS、结束只能经 report_done、循环检测/冷却、感知稳定等待 | **代码兜底**（这是给 LLM 的地基，不是替它做判断） |

**判断某段兜底代码是否健康的试金石——「契约收敛，补丁发散」：**

- ✅ **契约 / 不变量（可以加）**：数量**有限、稳定**，不随场景增长。例：「结束只能通过 report_done」「工具统一结构化返回码」「可判定事实由工具给 ground truth」「感知前等页面稳定」。这类兜底让 LLM 的输出有可靠的落点，反而**放大**了 LLM 的作用。
- ⚠️ **特例补丁（要警惕）**：针对某个具体异常写的 if/else，会**无限增长**（"如果在设置页反复点应用列表就……""如果输出里出现某字样就……"）。补丁越堆越多，是 approach 有问题的信号——正确解法通常是**换更强的模型 / 改 prompt / 改工具设计**，而不是再加一个补丁。

> 一句话：**代码负责"稳"（契约、事实、基础设施），LLM 负责"活"（理解、决策、判断）。** 凡是让"稳"的部分收敛、让 LLM 更放得开的兜底，就做；凡是让代码陷入无限特例补丁的，就停下来换思路。

## 1) 分类标准

两个维度独立打分，再合成分级：

- **紧急性**：是否正在造成实际失败/痛点。
  - 高 = 当前就在导致用例失败（如「找不到控件」）
  - 中 = 影响质量/稳定性但不必现在停下手上的事
  - 低 = 暂无实际影响，属预防性/长期性
- **必要性**：修复对正确性/稳定性/可维护性的根本重要程度。
  - 高 = 关系到结果正确性或核心稳定性
  - 中 = 明显改善但有替代/可容忍
  - 低 = 锦上添花

合成分级：

| 分级 | 含义 | 判定 |
|---|---|---|
| **P0 立即做** | 正在造成失败、且必须修 | 紧急高 × 必要高 |
| **P1 尽快规划** | 对正确性/稳定性关键，但不是当前唯一痛点 | 必要高 × 紧急中 |
| **P2 按需推进** | 明显改善，成本可控，非阻塞 | 必要中 × 紧急中/低 |
| **P3 长期/可选** | 预防性、可维护性、架构优化 | 必要或紧急偏低 |
| **✅ 已完成** | 旧文档计划项，经代码核对确认已落地 | — |

## 4) P0 — 立即做（正在造成「找不到控件」）

这四项共同根因是**感知（perceive）与执行（click）之间的时序竞态**：`agent_node` 采样页面 A、构造带 `[index]` 的列表给 LLM，LLM 思考数秒后返回 `click(index=k)`，而 `click()` 内部**重新感知一次**——两次快照顺序/数量一旦不同（动画、异步加载、Toast、状态栏刷新），`index=k` 就点错/越界。

- **R1 感知稳定等待**：`perceive()` 采样前先轮询 `dump_hierarchy()` 的 hash，连续 2~3 次不变或超时（~1.5s）再返回，过滤中间态。**性价比最高。**
- **R2 定位优先级 index→rid/path**：`agent.txt` 改为 `rid > path_contains > class_name > index`（rid/path 跨快照稳定，index 最易漂）；`click()` 里给了 rid/path 就忽略 index。
- **R3 click 失败有限重试**：定位失败（非明确 NOT_FOUND）时 `sleep(0.5)→重感知→再定位`，最多 2~3 次。
- **R4 元素列表上限提高**：从 25 提到 60~80，超限按「与目标/verification 文本相关性」排序后再截断，别简单 break；`index` 编号与最终展示列表严格对应。（代码位置确认：`agents/nodes.py` 的 `agent_node` 仍有 `if len(lines) > 25: break`。）

> **实测（2026-07-11 运行日志）**：R4 的截断已被印证，但表现不是"找不到控件"而是**效率代价**——目标 `index=34`（>25）时不在注入的 `page_info` 里，模型只能反复调 `get_screen_info`（该工具全量、不截断）来补看，本次 35 轮里 `get_screen_info` 占了 **13 次**。所以修 R4（放开上限）能同时减少"看不到→再 get_screen_info"的往返**和** token 膨胀（见 §6 O2）。这也解释了：为何本次用 index 定位（R2 未复现漂移）却依旧成功——模型靠 get_screen_info 拿到了最新全量列表。

> 建议先做 R1 + R2 + R3，直接命中竞态根因，通常立竿见影；做完用真机回放集验证失败率再推 R4。

## 5) P1 — 尽快规划（正确性/稳定性关键）

- **R6 中文 rerank bug**：`_experience_relevance` 用 `.split()` 按空格分词，中文无空格 → 整句变一个 token → relevance 恒 0 → 排序失效。改用字符级 2-gram 或直接复用向量 distance（可借 §7 的 `EnsembleRetriever`）。**成本极低的确定性 bug 修复。**
- **M1 提示过多（注意：不要走向"特例补丁"）**：现在每轮注入 8 类情境提示，本质是**迁就弱模型的软补丁**——但正确解法**不是**把它们翻译成"代码硬拦截"（那会走向 §0 警惕的发散型补丁）。方向应是**减少**这类情境提示，靠 **更强的模型 + 更干净的工具契约（M2/L1）** 把它们变得不必要；只保留**通用基础设施**（循环检测 / 冷却，如已有的 `COOLDOWN_SKIP`——任何 agent 都需要防死循环，属稳定契约而非特例补丁）。同时限制单轮注入提示数，多条命中只留优先级最高 1 条。
  - **落地（已完成）**：`agent_node` 把 4 类预调用软提示（`FINALIZATION_HINT` / `KNOWLEDGE_QUERY_REQUIRED` / `SELF_DOUBT_HINT` / `APP_SWITCH_HINT`）改为**先收集候选、按优先级排序后单轮只注入最高 1 条**（优先级：FINALIZATION > KNOWLEDGE_QUERY > SELF_DOUBT > APP_SWITCH）。有状态标记（`_finalization_hint_injected`/`_knowledge_query_hint_injected`）仅在**实际注入**时置位，未选中的下一轮仍可再命中。config 开关 `single_hint_per_turn`（默认 `True`）可关闭回退旧行为（全部注入）。
  - **说明**：这是**减少注入量**（收敛方向），不是把提示翻译成硬拦截；符合 §0。属行为类改动（改变了模型每轮可见的提示），建议真机回放确认。
- **M2 终止靠正则**：用**结构化契约**取代文本猜测——`report_done` 定为唯一合法终止，配合 §7 `tool_choice` 在收尾阶段强制工具调用。注意：那条 `DONE:` 正则本身才是补丁（因弱模型有时不调工具），此项是**去掉一个补丁**、让 LLM 用好自己的结构化能力，而非新增兜底。
  - **落地（安全版，已完成）**：`agent.txt` 输出格式改为「终止契约」——明确 **report_done 是结束任务的唯一合法方式**，「只用自然语言说完成了/done 都不算结束」，且「每一步必须调用恰好一个工具」。这是纯提示层的**契约强化**（收敛型、零风险、可回滚），直接降低弱模型「说完成了但不调工具→空转到预算耗尽」的概率。
  - **暂缓（需真机调参）**：① **暂不删除** `_detect_termination` 的 `DONE:`/`ABORT:` 正则——保留为安全兜底网，避免模型偶发不调工具时直接回退到「空转耗尽预算」；待真机确认新契约稳定后再评估下线。② **暂不做** `tool_choice` 强制 report_done：在收尾阶段强制它会有**过早终止**风险（模型可能还差 1–2 步操作），而「验证全过」的正常收尾 `route_after_agent` 已自动路由到 reporter 终止，`tool_choice` 强制的有效窗口窄、收益低于风险。此项留待 L1（工具契约结构化）一并做真机验证。
- **M4 验证靠模型**：给 LLM **可靠的验证工具**，而不是替它做判断。文本/元素存在性这类**确定性事实**，让 `assert_page_contains`/`assert_element_exists` 直接给 ground truth（结果直接写入 `_verifications`，省掉模型二次主观上报，减少假阳性）；**颜色/布局/Toast/"是否符合预期"等语义判断仍交给 LLM + 视觉**。是"LLM 用好工具"，不是"代码接管验证"。
  - **落地（已完成）**：① `assert_page_contains`/`assert_element_exists` 每次调用把 `{text, kind, result}` 记入 `ToolContext._deterministic_checks`；② `assert_verification` 反查最近一条与 `condition` 匹配（保守子串包含）的确定性核实作为 ground truth，写入 `detail`（因 `_merge_goal_verification_results` 只保留 `key/item/result/detail/screenshot` 白名单字段，证据折叠进 `detail` 才能传到 reporter）；③ **默认「仅证据」**——一致则标 `[代码核实=PASS]`，冲突则标 `[⚠️代码核实=FAIL，与模型判定冲突]` 但**不改判定**（零风险、可回滚、不破坏现有 verdict）；④ 配置开关 `deterministic_verification_override`（默认 `False`）置 `True` 时开启**硬核实**——冲突按代码结果修正 `result`。⑤ `agent.txt` 增补引导：可字面判定的验证项先调确定性断言再 `assert_verification`。
  - **说明**：override 默认关闭，是因为 `condition→断言` 的映射是模糊匹配，硬翻转判定需真机验证匹配精度后再开；当前「仅证据」模式已能把冲突暴露给报告/人工复核，安全地降低假阳性。
- **L1 工具契约标准化**：click/verify 等核心工具统一结构化返回 `code(OK/AMBIGUOUS/NOT_FOUND/SAFETY_BLOCKED)/message/hints/evidence`。这是**给 LLM 的稳定地基**——LLM 拿到明确的结构化结果就能自己决定下一步，无需代码去猜它的意图；也是"先把契约做稳、再放开 LLM/下线 legacy"的前提。
  - **落地（已完成，向后兼容版）**：新增 `tools/results.py` —— 有限稳定的状态码词表（`OK/NOT_FOUND/AMBIGUOUS/ERROR/NEEDS_HUMAN/PASS/FAIL`）+ 构造器 `make_result(status, message, evidence)` + 解析器 `parse_status`/`parse_evidence`/`is_fallback_output`。返回格式 `"<STATUS>: <message> || k=v; k=v"`（证据分隔符 `' || '` 与 click 既有的 `' | '` 快照拼接不冲突）。
    - `tools/click.py`：把散落的失败态**统一**——精确定位「未找到」→ `NOT_FOUND`、「N 个候选」→ `AMBIGUOUS`；`滑动后仍未找到`/`未找到可点击元素`/`long_press 未找到`/`未能点击` 全部加规范前缀。成功态与既有断言（`已点击`/`AMBIGUOUS:`/`PASS`/`FAIL`）保持不变，现有测试不破。
    - `agents/loop_control.py`：`_resolve_click_match_mode` 改读 `parse_status`（识别 `AMBIGUOUS`），`_resolve_click_fallback` 改按 `strategy=` **枚举**（`is_fallback_output`）判定，**不再 grep "fallback"/"ambiguous" 子串**；R3 重试条件由 `"未找到匹配元素" in err` 改为 `parse_status(err)==NOT_FOUND`（自然排除 AMBIGUOUS，不再误重试）。
    - **向后兼容**：`parse_*` 对无前缀的旧返回宽容（返回空），解析方回退到旧启发式——因此可增量迁移，未改到的工具不受影响。`agent.txt` 增补「工具返回状态」统一约定。
  - **说明**：改动改变了 LLM 可见的部分返回文案（失败态前缀），属行为类改动，建议真机回放确认模型对新状态码的响应；完整的 `hints/evidence` 富结构化（如 `content_and_artifact`）与 P3 legacy 下线可在此地基上后续推进。
- **M3 统一 OpenAI 兼容接入、去掉 zhipu 原生 SDK【已决策】**：不再维护 zhipu 原生 SDK 分支——**zhipu 及后续要用的多模态模型，统一通过 OpenAI 兼容端点走 LangChain `ChatOpenAI`**（`config.yaml` 里配 `base_url` 指向对应的 OpenAI 兼容地址即可）。收益：
  - **只剩一条 provider 路径** → `temperature` 可配、错误/token 统计天然对齐（不再有 openai/zhipu 两套不一致的问题）；
  - **直接解锁 `with_structured_output` / `with_fallbacks` / `tool_choice`**（见 §7）——不再有"zhipu 走原生 SDK 享受不到"的限制，因此 **M3 是 M2/M4/planner 结构化改造的前置，应在本批最先做**；
  - **代码更简单**：删掉 `agents/llm_runtime.py::_run_agent` 的 zhipu 分支、`llm/clients.py` 的 `ZhipuTextClient`/`ZhipuVLMClient`（原生 SDK 封装）、`requirements.txt` 的 `zhipuai` 依赖；`llm/multimodal.py` 的视觉调用同样收敛到 OpenAI 兼容多模态接口。
  - 契合 §0：这是"把 provider 接入收敛成单一稳定契约"，属**收敛型**改动，不是补丁。

> M2/M3/M4/L1 的共同点符合 §0 原则：**代码只负责"稳"的部分——结构化契约与可核实的事实（收敛、有限）；"活"的部分（找元素、决策、语义判断）继续放给 LLM。** 目标是让 LLM 通过 Agent 发挥更大作用，而不是用代码替它做判断。M1 则是反面提醒：**别把软补丁换成硬补丁**，那是发散型兜底，要靠更强模型 + 更干净契约来消解。

当前**唯一紧急**的是 P0 那组「找不到控件」——根因是感知/执行时序竞态，靠加稳定等待、把 index 换成 rid/path、失败重试、放开元素截断即可显著缓解。其余问题（模型依赖、RAG 质量、legacy 下线）遵循 §0 的指导原则：**代码只负责"稳"的地基——结构化契约、可核实的事实、循环检测这类通用基础设施（收敛、有限）；"活"的部分——找元素、决策、语义判断——继续放给 LLM，让它通过 Agent 发挥更大作用。** 判断准绳是「契约收敛、补丁发散」：凡是让 LLM 更放得开的兜底就做，凡是让代码陷入无限特例补丁的就换思路（更强模型 / 改 prompt / 改工具设计）。这与团队既有的「LLM-Native」路线一致（legacy 元素匹配正在下线），按 P1→P2→P3 分批推进即可，不必与 P0 抢工期。

**更深一层根因（回答「为啥旧 log 能点击试探+自判类型，改后不行」）**：核对最旧 good run（224128）——菜单 3 图标 label **同样都是"应用信息"**（感知问题一直存在），但旧 Agent 用的是 **label-only 语义点击** `click("应用信息")` → 走**语义路径**（`_find_best_element_with_known` 取**单个最佳**候选）→ `strategy=bounds` **直接点了**一个 → 观察效果 → 自判"这是添加到桌面"。**语义点击永远会"动手"**，所以"点击试探+观察"的探索策略成立。而改后我们用两处引导把模型推向**精确模式**：① `agent.txt`「优先 rid/path/class」；② AMBIGUOUS 提示「请追加 path_contains 或 rid」。精确模式对同名同类兄弟会**返回 AMBIGUOUS、拒绝动手** → 模型拿不到观察结果 → 只能反复堆参数 → 空转。**本质：我们把 LLM 从它擅长且有效的「语义点击→观察→判断」策略上挤了下来，推到一条对同名元素会死锁的严格路径**（正是 §0 警惕的"代码过度约束 LLM"）。
> **补充修复（本轮）**：AMBIGUOUS 提示改为**数据驱动的通用提示**——当候选 label 全相同（不限任何具体场景）时提示「用 index 选第几个」，而不是徒劳地「追加 path/rid」。这是**通用契约**（由实际候选计算，不含硬编码例子），配合 N2（index 在精确模式内可选位）让精确路径也能选中同类兄弟；label-only 语义点击的探索路径**始终保留**，模型可继续"点一个→观察→判断"。
>
> **原则纠偏（应 review 要求）**：先前 agent.txt 里写过带具体例子（"应用信息"/"system_shortcuts_container"）的精确点击指引——这违反「不针对特定例子加 patch」原则，且过度规定"优先 rid/path、index 垫底"的死板优先级反而把模型带偏。已**回退为通用表述**：确定性 disambiguation 交给代码（N2/label 收窄/同名提示），提示词只给 LLM 通用latitude，不再规定死板优先级、不举具体例子。

**根因认识**：`assert_page_contains` 的 **PASS 可信**（文字确实在），**FAIL 不可信**（文字未匹配 ≠ 元素不存在——图标/canvas/无 content-desc）。此前 M4 把两者对称当 ground truth，才对图标 UI 误报「冲突」、把成功压成 inconclusive。这是「代码越权否定 LLM」——违反 §0（代码只在权威处兜底）。

**方案 A 落地（本轮）**：
1. **`tools/verify.py` M4 非对称**：`assert_page_contains`/`assert_element_exists` **PASS = 权威 ground truth**（确认，冲突时可 override）；**FAIL = 不权威**——只在「模型也判 failed」时作弱佐证，**model=passed 时完全不标记、不否定、不 override**（图标场景不再误判冲突）。`override` 也只对 PASS 生效。
2. **`agent.txt`（通用）**：可字面判定项 PASS→据此 passed；**但 FAIL 不代表验证失败**，图标/无文字元素改用**行为效果**验证（执行动作+观察结果变化），不因文字断言 FAIL 就报 failed 或卡住。
> 符合 §0：把 M4 从「文字缺失即否定」收回到「只确认能确定的（PASS）」；图标是什么、点了有没有生效交给 LLM 观察判断。全通用、无具体例子。预期：v2/v3 不再假冲突、v1 由行为效果判 passed → verdict 回 passed。
