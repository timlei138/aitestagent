## 1) 分类标准

两个维度独立打分，再合成分级：

- **紧急性**：是否正在造成实际失败/痛点。
  - 高 = 当前就在导致用例失败（如「找不到控件」）
  - 中 = 影响质量/稳定性但不必现在停下手上的事
  - 低 = 暂无实际影响，属预防性/长期性
- **必要性**：修复对正确性/稳定性/可维护性的根本重要程度。
  - 高 = 关系到结果正确性或核心稳定性
  - 中 = 明显改善但有替代/可容忍
  - 低 = 锦上添花

合成分级：

| 分级 | 含义 | 判定 |
|---|---|---|
| **P0 立即做** | 正在造成失败、且必须修 | 紧急高 × 必要高 |
| **P1 尽快规划** | 对正确性/稳定性关键，但不是当前唯一痛点 | 必要高 × 紧急中 |
| **P2 按需推进** | 明显改善，成本可控，非阻塞 | 必要中 × 紧急中/低 |
| **P3 长期/可选** | 预防性、可维护性、架构优化 | 必要或紧急偏低 |
| **✅ 已完成** | 旧文档计划项，经代码核对确认已落地 | — |

## 4) P0 — 立即做（正在造成「找不到控件」）

这四项共同根因是**感知（perceive）与执行（click）之间的时序竞态**：`agent_node` 采样页面 A、构造带 `[index]` 的列表给 LLM，LLM 思考数秒后返回 `click(index=k)`，而 `click()` 内部**重新感知一次**——两次快照顺序/数量一旦不同（动画、异步加载、Toast、状态栏刷新），`index=k` 就点错/越界。

- **R1 感知稳定等待**：`perceive()` 采样前先轮询 `dump_hierarchy()` 的 hash，连续 2~3 次不变或超时（~1.5s）再返回，过滤中间态。**性价比最高。**
- **R2 定位优先级 index→rid/path**：`agent.txt` 改为 `rid > path_contains > class_name > index`（rid/path 跨快照稳定，index 最易漂）；`click()` 里给了 rid/path 就忽略 index。
- **R3 click 失败有限重试**：定位失败（非明确 NOT_FOUND）时 `sleep(0.5)→重感知→再定位`，最多 2~3 次。
- **R4 元素列表上限提高**：从 25 提到 60~80，超限按「与目标/verification 文本相关性」排序后再截断，别简单 break；`index` 编号与最终展示列表严格对应。

> 建议先做 R1 + R2 + R3，直接命中竞态根因，通常立竿见影；做完用真机回放集验证失败率再推 R4。

## 5) P1 — 尽快规划（正确性/稳定性关键）

- **R6 中文 rerank bug**：`_experience_relevance` 用 `.split()` 按空格分词，中文无空格 → 整句变一个 token → relevance 恒 0 → 排序失效。改用字符级 2-gram 或直接复用向量 distance（可借 §7 的 `EnsembleRetriever`）。**成本极低的确定性 bug 修复。**
- **M1 提示过多**：把「软提示」逐步升级为「代码硬控制」（参考已有 `COOLDOWN_SKIP` 直接拦截的做法）；限制单轮注入提示数，多条命中只留最高优先级 1 条。
- **M2 终止靠正则**：`report_done` 定为唯一合法终止；检测到「输出完成语义但没调工具」时代码强制注入要求下一步必须调 `report_done`。可配合 §7 `tool_choice`。
- **M4 验证靠模型**：文本/元素存在性这类可判定项，让 `assert_page_contains`/`assert_element_exists` 结果**直接**写入 `_verifications`，不再让模型二次上报（减少假阳性）；只有颜色/布局/Toast 才走模型+视觉。
- **L1 工具契约标准化**：click/verify 等核心工具统一结构化返回 `code(OK/AMBIGUOUS/NOT_FOUND/SAFETY_BLOCKED)/message/hints/evidence`，这是「先结构化输出稳定、再下线启发式」的前提，也让 M/L 系列改动有稳定契约可依赖。

> M1/M2/M4/L1 都指向同一主题：**把该由代码确定性完成的判断从「靠模型遵循提示」迁移到「代码兜底」**——这与 migration 文档的「LLM-Native」方法论一致，只是这次迁的是"补确定性兜底"而非"删启发式"。

## 6) P2 — 按需推进（明显改善，非阻塞）

- **R5 截图写盘**：改为按需/异步落盘，降单步延迟，间接缩小 R1 竞态窗口。
- **M3 provider 不对等**：`temperature` 提到 `config.yaml` 可配（建议 0~0.1）；zhipu 与 openai 的温度、`tool_call_400` 统计对齐（最好统一收敛到 LangChain `ChatOpenAI`，见 §7）。
- **G1 两通道口径统一**：抽统一的 `retrieve_knowledge(app_package, query, page_sig)`，`_rag_ctx`（自动注入）与 `query_app_knowledge`（工具）都走它，只是 top_k 不同，保证召回/去重/rerank 单一来源。（对应 rag_data_quality 的「统一查询入口」）
- **G2 规则不排序**：`query_curated_rules` 按语义相关性（向量 `query()`）或至少 `quality_score`/`last_verified_at` 排序后再截断——它是 planner hints 的第一优先级，不该「随便拿几条」。
- **G3 quality 门槛丢数据**：缺 `quality_score` 的历史给默认分（如 0.5），或过滤后为空则放宽兜底，避免「有数据但查不到」。
- **L5 指标门禁**：`match_mode`/`fallback_used`/`rag_*` 埋点已落地，补一个 gate_check 脚本把 `fuzzy_click_rate`/`ambiguous_rate` 等阈值跑成闭环，为后续 legacy 下线提供数据依据。

## 8) P3 — 长期/可选

- **L2 legacy 启发式点击代码下线**（Phase 2/3，`_score_element`/`_rank_click_candidates`/多路 fallback 等 ~800 行）：可维护性优化，需先满足 migration §6 的下线门槛（精确模式占比、fallback 触发率达标）。

- **L4 规则单一来源（policy 化）**：prompt 与运行时规则统一到 machine-readable policy。
- **L6 `save_curated_rule` limit=500 边界**：curated_rule 超 500 条可能漏重复，改 metadata 内嵌 `dedupe_key` 从过滤层去重。

## 11) 一句话总结

当前**唯一紧急**的是 P0 那组「找不到控件」——根因是感知/执行时序竞态，靠加稳定等待、把 index 换成 rid/path、失败重试、放开元素截断即可显著缓解。其余问题（模型依赖、RAG 质量、legacy 下线）本质是同一个方法论：**把该由代码确定性完成的判断，从「靠模型遵循提示」迁移到「代码兜底」**，与团队既有的「LLM-Native」路线一致，按 P1→P2→P3 分批推进即可，不必与 P0 抢工期。
